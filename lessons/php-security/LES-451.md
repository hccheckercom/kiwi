# Bug: Public REST Endpoint Without Rate Limiting

## Problem

REST endpoints using `'permission_callback' => '__return_true'` are accessible by anyone — no auth required. Without rate limiting, attackers can spam the endpoint to:
- Insert garbage data (analytics events, fake orders)
- Exhaust database resources (DDoS via INSERT flood)
- Generate thousands of order numbers / QR codes

**Example (BAD):**
```php
register_rest_route( 'wezone/v1', '/marketplace/orders', array(
    'methods'             => 'POST',
    'callback'            => array( $ctrl, 'create_order' ),
    'permission_callback' => '__return_true',  // anyone can POST
) );
// No rate limit → attacker can create 1000 orders/minute
```

## Fix

Add IP-based rate limiting via WordPress transient:

```php
private function check_rate_limit( string $action, int $max = 10, int $window = 60 ): bool {
    $ip  = $this->get_client_ip();
    $key = 'wz_rl_' . $action . '_' . md5( $ip );
    $count = (int) get_transient( $key );
    if ( $count >= $max ) {
        return false;
    }
    set_transient( $key, $count + 1, $window );
    return true;
}

public function create_order( WP_REST_Request $request ): WP_REST_Response {
    if ( ! $this->check_rate_limit( 'checkout', 5, 300 ) ) {
        return new WP_REST_Response( array( 'error' => 'Quá nhiều yêu cầu.' ), 429 );
    }
    // ... rest of logic
}
```

## Impact

- Without rate limit: bot can create thousands of fake orders, spam analytics
- With rate limit: max 5 orders per IP per 5 minutes
