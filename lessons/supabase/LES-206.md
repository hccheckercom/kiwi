## Bad
```sql
-- 036_remove_legacy_fields.sql
ALTER TABLE products DROP COLUMN legacy_category;
ALTER TABLE products DROP COLUMN old_sku;
```
Destructive change with no rollback plan. If migration breaks production, data is lost.

## Good
```sql
-- 036_remove_legacy_fields.sql
-- Step 1: Rename instead of drop (reversible)
ALTER TABLE products RENAME COLUMN legacy_category TO _deprecated_legacy_category;
ALTER TABLE products RENAME COLUMN old_sku TO _deprecated_old_sku;
-- Rollback: ALTER TABLE products RENAME COLUMN _deprecated_legacy_category TO legacy_category;
```

## Why
Supabase migrations are forward-only by default. Destructive operations (DROP, RENAME) without a rollback comment or backup strategy mean you cannot undo mistakes in production. Rename columns instead of dropping them; keep for 1 release cycle before actual removal.
