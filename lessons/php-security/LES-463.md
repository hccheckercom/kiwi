## Problem

Interpolating table names like `{$daily_table}` inside prepared statements with `phpcs:disable` comment does not eliminate SQL injection risk. If `$wpdb->prefix` is compromised (via plugin vulnerability, config injection, or database breach), attackers can inject arbitrary SQL through the table name.

## Bad

```php
$daily_table = $wpdb->prefix . 'wezone_analytics_daily';

// phpcs:disable WordPress.DB.PreparedSQL.InterpolatedNotPrepared
$rows = $wpdb->get_results(
    $wpdb->prepare(
        "SELECT * FROM {$daily_table} WHERE date >= %s",
        $from
    )
);
// phpcs:enable
```

**Risk:** If `$wpdb->prefix` becomes `wp_'; DROP TABLE wp_users; --`, the query becomes:
```sql
SELECT * FROM wp_'; DROP TABLE wp_users; -- wezone_analytics_daily WHERE date >= '2026-05-01'
```

## Good

**Option 1: Validate table name against allowlist**

```php
$allowed_tables = [
    'wezone_analytics_daily',
    'wezone_analytics_products',
];
$table_suffix = 'wezone_analytics_daily';

if (!in_array($table_suffix, $allowed_tables, true)) {
    return new \WP_Error('invalid_table', 'Invalid table name');
}

$daily_table = $wpdb->prefix . $table_suffix;

// Now safe — table name validated
$rows = $wpdb->get_results(
    $wpdb->prepare(
        "SELECT * FROM {$daily_table} WHERE date >= %s",
        $from
    )
);
```

**Option 2: Use `$wpdb->tables` for core tables**

```php
// For core WP tables only
$rows = $wpdb->get_results(
    $wpdb->prepare(
        "SELECT * FROM {$wpdb->posts} WHERE post_type = %s",
        'product'
    )
);
```

## Apply

- Any custom table name interpolation MUST validate suffix against allowlist
- Add allowlist constant at top of class: `private const ALLOWED_TABLES = ['table1', 'table2'];`
- Validate before concatenating with `$wpdb->prefix`
- Return `WP_Error` if validation fails
