## Pattern

Methods that modify balance (adjust, revoke, expire) call `log_transaction()` + `update_balance()`
without wrapping in a transaction. Concurrent requests can interleave, causing incorrect balance.

## Bad

```php
public function adjust( int $user_id, int $points, string $note = '' ): void {
    $balance = max( 0, $this->repo->get_balance( $user_id ) + $points );
    // No transaction — concurrent adjust calls can read stale balance
    $this->repo->log_transaction( $user_id, 'adjust', $points, $balance, null, $note );
    $this->repo->update_balance( $user_id, $this->tiers );
}
```

## Good

```php
public function adjust( int $user_id, int $points, string $note = '' ): void {
    global $wpdb;
    $wpdb->query( 'START TRANSACTION' );

    $balance = max( 0, $this->repo->get_locked_balance( $user_id ) + $points );
    $ok = $this->repo->log_transaction( $user_id, 'adjust', $points, $balance, null, $note );

    if ( ! $ok ) {
        $wpdb->query( 'ROLLBACK' );
        return;
    }

    $wpdb->query( 'COMMIT' );
    $this->repo->update_balance( $user_id, $this->tiers );
}
```

## Why

Without transactions, two concurrent admin adjustments can both read balance=100,
both add 50, and both write balance=150 instead of the correct 200. The `get_locked_balance()`
with `FOR UPDATE` prevents this by serializing concurrent reads.
