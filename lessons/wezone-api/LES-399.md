## Pattern

`OrderEngine::create()` uses `current_time('mysql', true)` while `OrderStateMachine::transition()` and `OrderHistory::record()` use `gmdate('Y-m-d H:i:s')`. Both produce UTC, but:
1. `current_time()` adds WordPress filter overhead
2. Mixing styles makes code review harder — reviewer must verify both produce same timezone
3. If WordPress `pre_get_current_time` filter exists, results could diverge

## Wrong

```php
// OrderEngine::create()
'created_at' => current_time('mysql', true),  // WordPress function

// OrderStateMachine::transition()
'updated_at' => gmdate('Y-m-d H:i:s'),       // PHP native

// OrderHistory::record()
'created_at' => gmdate('Y-m-d H:i:s'),       // PHP native
```

## Correct

```php
// All Order classes — use gmdate() consistently
'created_at' => gmdate('Y-m-d H:i:s'),
'updated_at' => gmdate('Y-m-d H:i:s'),
```

## Why

Consistency prevents timezone bugs during DST transitions or when WordPress timezone settings change. One function, one behavior, zero ambiguity.
