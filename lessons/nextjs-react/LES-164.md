## Bad
```js
module.exports = {
  apps: [{
    name: 'wezone',
    script: 'server.js',
    instances: -1,        // All CPU cores
    exec_mode: 'cluster', // Multiple instances bind same port → EADDRINUSE
  }]
}
// Result: crash loop — PM2 restarts, crashes again, restarts...
```

## Good (pattern from ecosystem.config.js)
```js
module.exports = {
  apps: [{
    name: 'wezone',
    script: 'server.js',
    cwd: '/opt/wezone',
    instances: 1,         // Single instance
    exec_mode: 'fork',    // Fork mode — no port conflict
    env: {
      NODE_ENV: 'production',
      PORT: 3000,
      HOSTNAME: '127.0.0.1',
    },
    // Recovery
    restart_delay: 2000,
    max_restarts: 10,
    min_uptime: '5s',
    // Graceful shutdown
    kill_timeout: 5000,
    listen_timeout: 15000,
    wait_ready: false, // server.js doesn't send process.send('ready')
  }]
}
```

## Also: wait_ready trap
```js
// BAD: wait_ready: true with Next.js standalone
// Next.js server.js never calls process.send('ready')
// PM2 waits listen_timeout then kills → infinite restart loop

// GOOD: wait_ready: false
```

## Why
Real pattern from ecosystem.config.js. Next.js standalone server.js doesn't support PM2 cluster mode — multiple instances try to bind the same port. Also: wait_ready: true causes crash loop because server.js never sends the 'ready' signal. Fork mode with 1 instance is the correct config for Next.js standalone.
