# Bug: $GLOBALS as Runtime Flag

## Problem

Using `$GLOBALS['_wz_*']` as a boolean flag to prevent re-entrance or signal state between methods. Issues:

1. **Not thread-safe** — WordPress can run multiple requests concurrently; `$GLOBALS` is per-request only
2. **Not testable** — unit tests must manipulate global state
3. **Namespace collision** — other plugins could use the same key
4. **Implicit coupling** — reader must search entire codebase to find who sets/reads the flag

**Example (BAD):**
```php
// In check_speed()
if (empty($GLOBALS['_wz_score_speed_check_running'])) {
    wp_schedule_single_event(time(), 'wz_score_speed_check');
}

// In run_speed_check()
$GLOBALS['_wz_score_speed_check_running'] = true;
```

## Fix

Use a static property on the class:

```php
class StoreScorer {
    private static bool $speed_check_running = false;

    private function check_speed(): array {
        if (!self::$speed_check_running && !wp_next_scheduled('wz_score_speed_check')) {
            wp_schedule_single_event(time() + 5, 'wz_score_speed_check');
        }
        // ...
    }

    public function run_speed_check(): void {
        self::$speed_check_running = true;
        // ...
    }
}
```

Or for cross-request state, use a transient lock (see LES-447).

## Impact

- Cleaner architecture, testable code
- No risk of global namespace collision
