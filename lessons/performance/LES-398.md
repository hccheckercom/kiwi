## Pattern

`OrderStateMachine::transition()` is public and can be called directly (not just via `OrderEngine::updateStatus()`). When called directly, the order cache is not invalidated — `OrderEngine::get()` returns stale status for up to 300 seconds.

## Wrong

```php
// OrderStateMachine::transition()
$wpdb->update($wpdb->prefix . 'wz_orders', ['status' => $to], ['id' => $order_id]);
// ← no cache invalidation
do_action(Hooks::ORDER_STATUS_CHANGED, $order_id, $from, $to);
```

## Correct

```php
$wpdb->update($wpdb->prefix . 'wz_orders', ['status' => $to], ['id' => $order_id]);
wp_cache_delete("wz_order_{$order_id}", 'wezone_orders');
do_action(Hooks::ORDER_STATUS_CHANGED, $order_id, $from, $to);
```

## Why

Stale cached order status means: customer sees "pending" when payment is complete, admin dashboard shows wrong counts, webhook handlers make decisions on outdated state. Cache TTL is 300s — 5 minutes of wrong data.
