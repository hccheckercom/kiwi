# JSON field handling in repositories

## Problem
Database columns storing JSON must be properly encoded/decoded. Missing JSON handling causes type errors (string vs array) and data corruption.

## Pattern
1. **On read**: Decode JSON string to array with fallback
2. **On write**: Encode array to JSON string
3. **Handle both cases**: Field might already be string (from DB) or array (from code)

## Bad Examples

```php
// BAD: No JSON decoding on read
public function get( int $id ): ?array {
    global $wpdb;
    $row = $wpdb->get_row(
        $wpdb->prepare( "SELECT * FROM {$wpdb->prefix}wz_variations WHERE id = %d", $id ),
        ARRAY_A
    );
    return $row ?: null;
}
// Usage: $variation['attributes'] is string, not array → fatal error on array access
```

**Issue**: `attributes` column is JSON string, but code expects array.

```php
// BAD: No JSON encoding on write
public function create( int $product_id, array $data ): int {
    global $wpdb;
    $wpdb->insert( 
        $wpdb->prefix . 'wz_variations', 
        array(
            'product_id' => $product_id,
            'attributes' => $data['attributes'], // Array inserted as "Array"
        )
    );
    return (int) $wpdb->insert_id;
}
```

**Issue**: Array is converted to string "Array" instead of JSON.

## Good Examples

```php
// GOOD: Decode JSON on read with fallback
public function get( int $id ): ?array {
    global $wpdb;
    $row = $wpdb->get_row(
        $wpdb->prepare( "SELECT * FROM {$wpdb->prefix}wz_variations WHERE id = %d", $id ),
        ARRAY_A
    );
    
    if ( $row ) {
        $attrs = $row['attributes'];
        // Handle both string (from DB) and array (already decoded)
        $row['attributes'] = is_string( $attrs ) 
            ? ( json_decode( $attrs, true ) ?? array() ) 
            : ( is_array( $attrs ) ? $attrs : array() );
    }
    
    return $row ?: null;
}
```

```php
// GOOD: Encode JSON on write
public function create( int $product_id, array $data ): int {
    global $wpdb;
    
    $attrs = $data['attributes'] ?? array();
    unset( $data['attributes'] );
    
    $insert = array_merge(
        $this->sanitizeData( $data ),
        array(
            'product_id' => $product_id,
            'attributes' => wp_json_encode( $attrs ), // Encode to JSON string
            'created_at' => gmdate( 'Y-m-d H:i:s' ),
        )
    );
    
    $wpdb->insert( $wpdb->prefix . 'wz_variations', $insert );
    return (int) $wpdb->insert_id;
}
```

```php
// GOOD: Handle both string and array on update
public function update( int $id, array $data ): bool {
    global $wpdb;
    
    if ( isset( $data['attributes'] ) && ! is_string( $data['attributes'] ) ) {
        $data['attributes'] = wp_json_encode( $data['attributes'] );
    }
    
    $result = $wpdb->update(
        $wpdb->prefix . 'wz_variations',
        $this->sanitizeData( $data ),
        array( 'id' => $id )
    );
    
    return $result !== false;
}
```

## Detection Rules
- Repository `get()` method returns row with JSON column without decoding
- Repository `create()` inserts array into JSON column without `wp_json_encode()`
- Repository `update()` updates JSON column without checking if encoding needed
- Missing fallback `?? array()` after `json_decode()`

## Why This Matters
- **Type safety**: Prevents "Cannot use string as array" fatal errors
- **Data integrity**: Prevents "Array" string being stored instead of JSON
- **Consistency**: Ensures JSON fields are always arrays in PHP code
