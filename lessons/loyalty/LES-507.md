## Bad
```php
public function redeem_points( int $user_id, int $points ): array {
    $balance = $this->repo->get_balance( $user_id );
    if ( $points > $balance ) {
        return ['error' => 'Not enough points'];
    }
    
    $this->repo->log_transaction( $user_id, 'redeem', -$points, $balance - $points );
    // Race condition: 2 concurrent redeems both pass balance check
}
```

## Good
```php
public function redeem_points( int $user_id, int $points ): array {
    global $wpdb;
    $wpdb->query( 'START TRANSACTION' );
    
    $balance = $this->repo->get_locked_balance( $user_id ); // SELECT ... FOR UPDATE
    if ( $points > $balance ) {
        $wpdb->query( 'ROLLBACK' );
        return ['error' => 'Not enough points'];
    }
    
    $ok = $this->repo->log_transaction( $user_id, 'redeem', -$points, $balance - $points );
    if ( ! $ok ) {
        $wpdb->query( 'ROLLBACK' );
        return ['error' => 'System error'];
    }
    
    $wpdb->query( 'COMMIT' );
    return ['success' => true];
}
```

## Why
User with 100 points sends 2 concurrent redeem requests for 100 points each. Both read balance=100, both pass check, both deduct 100 -> final balance=-100. Must use SELECT FOR UPDATE to lock row.

## Grep
`function\s+redeem.*?\{.*?get_balance\(.*?\)(?!.*FOR\s+UPDATE|get_locked_balance)` — in `**/*.php`
