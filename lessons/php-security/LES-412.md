## Pattern

WordPress option names constructed via string interpolation without sanitization allow injection attacks. Even when the variable is validated against a whitelist with `in_array()`, the value itself is not sanitized before being used in the option name. This creates a gap where malicious input can inject SQL wildcards or special characters into option queries.

## Bug

```php
// SAI — $module validated but not sanitized
class Commerce {
    private const MODULES = array( 'products', 'cart', 'orders' );
    
    public static function disable( string $module ): void {
        // Whitelist check — looks safe?
        if ( ! in_array( $module, self::MODULES, true ) ) {
            return;
        }
        
        // But $module value itself is not sanitized!
        update_option( "wezone_commerce_{$module}_enabled", false, false );
    }
}

// Attack vector 1: SQL injection via LIKE wildcards
$malicious = "products%"; // matches products, products_v2, products_cache, etc.
if ( in_array( $malicious, Commerce::MODULES, true ) ) {
    // Fails — good
} else {
    // But if attacker controls the validation logic...
    update_option( "wezone_commerce_{$malicious}_enabled", false );
    // Deletes multiple options if using LIKE queries internally
}

// Attack vector 2: Option name collision
$malicious = "products'; DROP TABLE--";
// Even if in_array() blocks this, the pattern is still vulnerable
// if validation is bypassed or removed in future refactoring
```

**Hậu quả:**
- Option name injection → unintended options modified/deleted
- SQL wildcards in option names → mass deletion via LIKE queries
- Future refactoring removes validation → injection vulnerability exposed
- No defense-in-depth — validation is the only protection layer

## Fix

### Option 1: Sanitize with sanitize_key() (recommended)

```php
// ĐÚNG — sanitize before interpolation
class Commerce {
    private const MODULES = array( 'products', 'cart', 'orders' );
    
    public static function disable( string $module ): void {
        // Validate against whitelist
        if ( ! in_array( $module, self::MODULES, true ) ) {
            return;
        }
        
        // Sanitize before using in option name (defense-in-depth)
        $module = sanitize_key( $module );
        update_option( "wezone_commerce_{$module}_enabled", false, false );
    }
    
    public static function enable( string $module ): void {
        if ( ! in_array( $module, self::MODULES, true ) ) {
            return;
        }
        
        $module = sanitize_key( $module );
        update_option( "wezone_commerce_{$module}_enabled", true, false );
    }
}
```

### Option 2: Use array lookup instead of interpolation

```php
// ĐÚNG — no interpolation, no injection
class Commerce {
    private const OPTION_KEYS = array(
        'products' => 'wezone_commerce_products_enabled',
        'cart'     => 'wezone_commerce_cart_enabled',
        'orders'   => 'wezone_commerce_orders_enabled',
    );
    
    public static function disable( string $module ): void {
        if ( ! isset( self::OPTION_KEYS[ $module ] ) ) {
            return;
        }
        
        $option_name = self::OPTION_KEYS[ $module ];
        update_option( $option_name, false, false );
    }
}
```

### Option 3: Strict regex validation + sanitization

```php
// ĐÚNG — validate format + sanitize
class Commerce {
    public static function disable( string $module ): void {
        // Only allow lowercase letters and hyphens
        if ( ! preg_match( '/^[a-z-]+$/', $module ) ) {
            return;
        }
        
        // Sanitize as additional layer
        $module = sanitize_key( $module );
        update_option( "wezone_commerce_{$module}_enabled", false, false );
    }
}
```

## Áp dụng

**Mọi option name với variable interpolation PHẢI:**
1. Sanitize với `sanitize_key()` trước khi dùng trong option name
2. Hoặc: dùng array lookup thay string interpolation
3. Hoặc: validate format với strict regex + sanitize

**Defense-in-depth principle:**
- Validation (whitelist/regex) prevents invalid input
- Sanitization (sanitize_key) prevents injection even if validation is bypassed
- Both layers together = robust protection

**Applies to:**
- `update_option()`, `get_option()`, `delete_option()`
- `set_transient()`, `get_transient()`, `delete_transient()`
- `wp_cache_set()`, `wp_cache_get()`, `wp_cache_delete()`
- Any function accepting dynamic key/name parameters
