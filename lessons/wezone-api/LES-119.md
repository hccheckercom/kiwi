# LES-119 — wz_cart() Stale Data (3 Root Causes)

## Problem

Cart operations (qty change, remove) save to DB but subsequent reads return stale data — affecting both AJAX responses and full page renders (checkout, mini-cart).

## Affected Themes

- wezone-trunganh-v2 (documented in HANDOFF-CART-QTY-BUG.md)
- wezone-funilux (fixed 2026-05-19)

## 3 Root Causes

### RC1 — Duplicate rows in wp_wz_carts

`UNIQUE KEY idx_session_key` migration failed (pre-existing duplicates blocked ALTER TABLE). `$wpdb->replace()` requires UNIQUE KEY to upsert — without it, every `save()` INSERTS a new row instead of replacing the old one.

### RC2 — CartEngine::loadRow() missing ORDER BY

```php
// BUG — returns arbitrary row when duplicates exist
"SELECT * FROM wp_wz_carts WHERE session_key = %s AND expires_at > %s"

// FIX — always get most recent
"SELECT * FROM wp_wz_carts WHERE session_key = %s AND expires_at > %s ORDER BY updated_at DESC LIMIT 1"
```

This affects ALL pages that call `wz_cart()` — cart, checkout, mini-cart, etc.

### RC3 — Singleton object cache

`wz_cart()` is a static singleton. After `updateQty()` saves and calls `wp_cache_delete()`, the singleton's in-memory state may still be stale within the same PHP request.

## Symptoms

| Page | Symptom |
|------|---------|
| Cart (AJAX) | Order summary doesn't update after +/- |
| Checkout | Shows old qty and old total from stale DB row |
| Mini-cart | Badge count doesn't match |

## Fix Checklist

1. **Core fix:** Add `ORDER BY updated_at DESC LIMIT 1` to `CartEngine::loadRow()` query
2. **AJAX handlers:** Bypass singleton — query DB directly with `ORDER BY updated_at DESC LIMIT 1`
3. **Price recalc:** Use `wz_get_product()` for current catalog price, not stale `$item['price']` from cart
4. **Client-side:** Add `data-price` to cart item DOM, recalculate totals in JS using DOM prices
5. **Shipping config:** Add `data-free-threshold` and `data-shipping-fee` to order-summary container for JS calc

## Scanner Pattern

Detects any query on `wz_carts WHERE session_key` that lacks `ORDER BY updated_at DESC`. Flags as CRITICAL because duplicate rows cause stale reads.
