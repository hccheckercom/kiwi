# Bug Pattern

Firing a generic cache invalidation hook like `do_action('wezone/product/cache_invalidated')` without clearing specific cache keys can leave stale data in cache.

**Why:** Listeners may not know which specific products/sales to invalidate, leading to stale prices or availability data shown to users.

## Detection

```regex
do_action\(\s*['"][\w/_-]*cache[_-]?invalidat(?:e|ed)['"](?:\s*\)|,\s*\))
```

Match `do_action()` calls for cache invalidation without specific cache keys as parameters.

## Examples

### ❌ BAD: Generic cache invalidation

```php
public static function restore_prices( int $sale_id ): void {
    // Prices are dynamically filtered, so no stored prices need restoring.
    // But this hook doesn't tell listeners WHICH products to invalidate!
    do_action( 'wezone/product/cache_invalidated' );
}
```

**Problem:** Listeners don't know which products were in the sale, so they either:
1. Clear ALL product cache (expensive)
2. Don't clear cache at all (stale data)

### ✅ GOOD: Pass specific cache keys to invalidate

```php
public static function restore_prices( int $sale_id ): void {
    // Get all products in this sale
    $products = self::get_sale_products( $sale_id );
    $product_ids = array_column( $products, 'product_id' );
    
    // Fire hook with specific product IDs
    do_action( 'wezone/flash_sale/prices_restored', $sale_id, $product_ids );
    
    // Clear specific product caches
    foreach ( $product_ids as $product_id ) {
        wp_cache_delete( 'wezone_flash_sale_' . $product_id, 'wezone_flash_sale' );
        wp_cache_delete( 'product_' . $product_id, 'wezone_products' );
    }
}
```

### ✅ BETTER: Centralized cache invalidation helper

```php
public static function restore_prices( int $sale_id ): void {
    $products = self::get_sale_products( $sale_id );
    $product_ids = array_column( $products, 'product_id' );
    
    // Use centralized helper that knows all cache keys to clear
    if ( function_exists( 'wz_invalidate_product_cache' ) ) {
        foreach ( $product_ids as $product_id ) {
            wz_invalidate_product_cache( $product_id );
        }
    }
    
    // Fire hook for external listeners
    do_action( 'wezone/flash_sale/prices_restored', $sale_id, $product_ids );
}

// In wezone-core/src/Commerce/Product/Cache.php
function wz_invalidate_product_cache( int $product_id ): void {
    // Clear all known cache keys for this product
    wp_cache_delete( 'product_' . $product_id, 'wezone_products' );
    wp_cache_delete( 'wezone_flash_sale_' . $product_id, 'wezone_flash_sale' );
    wp_cache_delete( 'product_price_' . $product_id, 'wezone_prices' );
    wp_cache_delete( 'product_stock_' . $product_id, 'wezone_stock' );
    
    // Fire hook for external plugins
    do_action( 'wezone/product/cache_invalidated', $product_id );
}
```

## Fix Strategy

1. **Pass specific IDs** to cache invalidation hooks
2. **Clear cache keys directly** in the same function that fires the hook
3. **Use centralized helpers** (e.g., `wz_invalidate_product_cache()`) that know all cache keys
4. **Document cache keys** in function docblocks so future devs know what to clear
5. **Test cache invalidation** by checking product prices before/after sale ends
