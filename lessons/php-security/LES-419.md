# Bug Pattern

Cron callbacks that modify database state (INSERT/UPDATE/DELETE) must check capabilities, even though they run in system context.

**Why:** wp-cron.php can be triggered by unauthenticated HTTP requests. Without capability checks, attackers can manipulate cron timing to trigger state changes.

## Detection

```regex
add_action\(\s*['"][\w/_-]+['"],\s*array\([^)]+,\s*['"](\w+)['"]\s*\)\s*\);[\s\S]{0,500}public\s+function\s+\1\([^)]*\):\s*void\s*\{[\s\S]{0,1000}(?:INSERT|UPDATE|DELETE|wpdb->(?:insert|update|delete|query))
```

Match cron callbacks that perform DB writes without `current_user_can()` check.

## Examples

### ❌ BAD: Cron callback without capability check

```php
class FlashSaleScheduler {
    public function check_schedules(): void {
        global $wpdb;
        $table = $wpdb->prefix . 'wezone_flash_sales';
        
        // CRITICAL: No capability check before DB writes
        $wpdb->update( $table, array( 'status' => 'active' ), array( 'id' => $row['id'] ) );
    }
}

add_action( 'wezone_flash_sale_check', array( $scheduler, 'check_schedules' ) );
```

**Risk:** Attacker triggers wp-cron.php → activates sales prematurely.

### ✅ GOOD: Guard with capability check

```php
class FlashSaleScheduler {
    public function check_schedules(): void {
        // Guard: cron runs in system context, but verify we're in legitimate cron execution
        if ( ! wp_doing_cron() && ! current_user_can( 'manage_options' ) ) {
            wz_log( 'FlashSaleScheduler: unauthorized access attempt', 'security' );
            return;
        }
        
        global $wpdb;
        $table = $wpdb->prefix . 'wezone_flash_sales';
        $wpdb->update( $table, array( 'status' => 'active' ), array( 'id' => $row['id'] ) );
    }
}
```

### ✅ BETTER: Use nonce for cron verification

```php
class FlashSaleScheduler {
    public function check_schedules(): void {
        // Verify this is a legitimate cron execution
        if ( ! $this->verify_cron_context() ) {
            return;
        }
        
        global $wpdb;
        $table = $wpdb->prefix . 'wezone_flash_sales';
        $wpdb->update( $table, array( 'status' => 'active' ), array( 'id' => $row['id'] ) );
    }
    
    private function verify_cron_context(): bool {
        // wp_doing_cron() checks DOING_CRON constant set by wp-cron.php
        if ( wp_doing_cron() ) {
            return true;
        }
        
        // Fallback: check if user has admin capability (manual trigger from admin)
        if ( current_user_can( 'manage_options' ) ) {
            return true;
        }
        
        wz_log( 'FlashSaleScheduler: unauthorized cron trigger attempt', 'security' );
        return false;
    }
}
```

## Fix Strategy

1. Add `wp_doing_cron()` check at top of callback
2. Fallback to `current_user_can('manage_options')` for manual admin triggers
3. Log unauthorized attempts
4. Return early if verification fails
