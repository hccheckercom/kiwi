## Pattern

A `ServiceProvider` registers singletons in the DI container, but other classes
instantiate the same services with `new ClassName()` directly, bypassing the container.
This defeats the purpose of DI: no singleton guarantee, no testability, no override.

## Bad

```php
// ServiceProvider registers singleton
$this->container->singleton( PointsManager::class, fn($c) => new PointsManager(...) );

// Controller ignores the container
public function get_balance( $request ) {
    $manager = new PointsManager();  // ← new instance every call, not the singleton
    return $manager->get_balance( ... );
}
```

## Good

```php
class LoyaltyController {
    public function __construct(
        private PointsManager $manager,
        private TierManager $tier_mgr,
    ) {}

    public function get_balance( $request ) {
        return $this->manager->get_balance( ... );
    }
}

// Registration — resolve from container
$controller = $container->make( LoyaltyController::class );
```

## Why

1. Singleton guarantee broken — each `new` creates a separate instance with its own state
2. Testing impossible — can't mock/stub the dependency
3. Wasted memory — multiple instances of the same service
4. Configuration ignored — container may configure the service differently than default constructor
