## Bad
```ts
// Export ZIP contains real API key — anyone who downloads can access the store
zip.file('.env', `PLATFORM_API_KEY=${store.api_key}\nSUPABASE_URL=...`)
```

## Good (pattern from export route.ts)
```ts
// Generate .env.example with the key pre-filled (owner already has access)
// But name it .env.example so it's not accidentally committed to git
zip.file('.env.example', generateEnvFile(store.api_key, store.id, store.name, siteUrl))

// The generated .gitignore includes .env*
zip.file('.gitignore', 'node_modules/\n.next/\n.env*\n')
```

## Export security checklist
```
✓ Auth: requireStoreOwner(storeId) — only owner can export
✓ File naming: .env.example (not .env) — won't be committed
✓ .gitignore included in ZIP — prevents accidental commit
✓ No service role keys in export — only platform API key (scoped to this store)
✓ Cache-Control: no-store — ZIP not cached by CDN
✓ Content-Disposition: attachment — forces download, no inline rendering
```

## Why
Real pattern from platform export route. The ZIP contains the store's API key (needed for the template to work). Security measures: (1) only store owner can trigger export (requireStoreOwner), (2) file named .env.example (not auto-loaded), (3) .gitignore included, (4) no-store cache header prevents CDN caching of the ZIP.
