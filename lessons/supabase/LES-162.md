## Bad (multi-step insert — partial state if step 2 fails)
```ts
// Step 1: Create flash sale
const { data: sale } = await supabase.from('platform_flash_sales').insert({ ... }).select().single()

// Step 2: Create items — if this fails, sale exists without items (broken state)
const { error } = await supabase.from('platform_flash_sale_items').insert(
  items.map(i => ({ flash_sale_id: sale.id, ...i }))
)
// No rollback if step 2 fails!
```

## Good (RPC for atomic multi-table operations)
```sql
CREATE OR REPLACE FUNCTION create_flash_sale_with_items(
  p_store_id uuid, p_name text, p_starts_at timestamptz, p_ends_at timestamptz,
  p_items jsonb
) RETURNS uuid AS $$
DECLARE
  v_sale_id uuid;
BEGIN
  INSERT INTO platform_flash_sales (store_id, name, starts_at, ends_at)
  VALUES (p_store_id, p_name, p_starts_at, p_ends_at)
  RETURNING id INTO v_sale_id;

  INSERT INTO platform_flash_sale_items (flash_sale_id, store_id, product_id, sale_price, original_price)
  SELECT v_sale_id, p_store_id, (item->>'product_id')::uuid, (item->>'sale_price')::numeric, (item->>'original_price')::numeric
  FROM jsonb_array_elements(p_items) AS item;

  RETURN v_sale_id;
END;
$$ LANGUAGE plpgsql;
```

```ts
const { data, error } = await supabase.rpc('create_flash_sale_with_items', {
  p_store_id: store.id,
  p_name: body.name,
  p_starts_at: body.starts_at,
  p_ends_at: body.ends_at,
  p_items: JSON.stringify(body.items),
})
// Atomic — either all succeed or all rollback
```

## When to use RPC vs direct queries
| Scenario | Use |
|----------|-----|
| Single table CRUD | Direct `.from().insert/select/update` |
| Multi-table atomic | RPC function (PostgreSQL transaction) |
| Complex aggregation | RPC with `get_active_flash_sale` pattern |
| New column (schema cache stale) | RPC bypasses PostgREST cache |

## Why
Real pattern from flash-sales.handlers.ts. Multi-step inserts without transactions leave partial state when step N fails. PostgreSQL functions run in a single transaction — if any statement fails, everything rolls back. Also used for complex queries like `get_active_flash_sale` that JOIN multiple tables with business logic.
