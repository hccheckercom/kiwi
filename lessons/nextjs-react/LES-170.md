## Bad
```bash
# deploy.sh — .next cache from previous build may contain stale modules
cd /opt/wezone
git pull
npm run build  # Uses cached .next/cache → old code may persist
pm2 restart wezone
```

## Good (pattern from webstore-vn deploy)
```bash
cd /opt/wezone
git pull
rm -rf .next  # Clear ALL build cache — ensures fresh build
npm run build
pm2 restart wezone
```

## When stale cache causes real bugs
1. **Deleted module still imported**: Build succeeds (cached), but runtime crashes
2. **Renamed export**: Old cached chunk references old name → hydration error
3. **env var changed**: Build-time inlined NEXT_PUBLIC_ values are stale
4. **Middleware changes**: Old middleware.js cached → auth bypass

## Why
Real fix: commit 7f770ed. Next.js caches compiled pages, chunks, and webpack modules in .next/cache. After git pull with significant changes (deleted files, renamed exports, env changes), the cache may reference stale code. Production builds should always start fresh. Dev server handles this automatically with HMR, but production builds don't.
