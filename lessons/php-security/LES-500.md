## Bad
```php
// Plugin.php:394, 407, 413
register_rest_route( 'wezone/v1', '/marketplace/analytics/collect', array(
    'methods'             => 'POST',
    'callback'            => array( $collector, 'collect' ),
    'permission_callback' => '__return_true', // Public endpoint
) );
```

## Good
```php
// Option 1: Rate limiting inside callback
public function collect( \WP_REST_Request $request ): \WP_REST_Response {
    if ( ! $this->check_rate_limit( 'analytics', 30, 60 ) ) {
        return new \WP_REST_Response( array( 'ok' => false ), 429 );
    }
    // ... rest of logic
}

// Option 2: Nonce check for logged-in users
register_rest_route( 'wezone/v1', '/marketplace/orders', array(
    'methods'             => 'POST',
    'callback'            => array( $ctrl, 'create_order' ),
    'permission_callback' => function() {
        return is_user_logged_in() || wp_verify_nonce( $_SERVER['HTTP_X_WP_NONCE'] ?? '', 'wp_rest' );
    },
) );
```

## Why
`__return_true` permission callback allows unauthenticated access. Without rate limiting or nonce validation, endpoints are vulnerable to:
- Spam/abuse (analytics flooding)
- DoS attacks (resource exhaustion)
- CSRF (state-changing operations)

Rate limiting MUST be implemented inside the callback for public endpoints. State-changing operations (POST/PUT/DELETE) should require nonce or authentication.
