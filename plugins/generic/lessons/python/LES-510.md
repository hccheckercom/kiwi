# Bad Code

```python
def calculate_total(items, discount):
    total = sum(item['price'] for item in items)
    return total * (1 - discount)

def get_user(user_id):
    return db.query(User).filter(User.id == user_id).first()
```

# Good Code

```python
from typing import List, Dict, Optional

def calculate_total(items: List[Dict[str, float]], discount: float) -> float:
    total = sum(item['price'] for item in items)
    return total * (1 - discount)

def get_user(user_id: int) -> Optional[User]:
    return db.query(User).filter(User.id == user_id).first()
```

# Why

**Type hints improve:**
- **IDE support:** Autocomplete, refactoring, inline documentation
- **Early error detection:** Catch type mismatches before runtime
- **Code readability:** Function contracts are self-documenting
- **Maintainability:** Easier to understand function expectations

**Common issues without type hints:**
- Passing wrong types (string instead of int)
- Returning None when caller expects object
- Unclear function contracts
- Harder to refactor safely

**When to use:**
- All public functions and methods
- Functions with complex parameters
- Functions returning Optional values
- API endpoints and service layer functions

**Exceptions:**
- Simple private helpers with obvious types
- Test fixtures (optional)
- Migration scripts (temporary code)

# Grep

```bash
# Find functions without type hints
rg 'def\s+\w+\s*\([^)]*\)\s*:' --type py -g '!test_*.py' -g '!__init__.py'

# Find functions with parameters but no hints
rg 'def\s+\w+\s*\([^:)]+\)\s*:' --type py
```
