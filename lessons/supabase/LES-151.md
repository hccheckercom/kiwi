## When to use RPC vs direct query

| Scenario | Use | Why |
|----------|-----|-----|
| Normal CRUD | `.from('table').insert/select/update` | Type-safe, simple |
| New column just added | `.rpc('insert_xxx', params)` | Bypasses stale schema cache |
| Complex transaction | `.rpc('process_order', params)` | Atomic, no partial state |
| Generated columns (order_code) | `.rpc('create_order', params)` | Trigger may not exist in prod |

## Bad
```ts
// After adding gmc_business_id column via migration
const { error } = await supabase.from('orders').insert({
  order_code: code,
  gmc_business_id: businessId, // PostgREST cache doesn't know this column yet
})
// Error: "Could not find column 'gmc_business_id'"
```

## Good
```sql
-- Create RPC function in migration
CREATE OR REPLACE FUNCTION insert_gmc_order(p_code text, p_business_id uuid)
RETURNS uuid AS $$
  INSERT INTO orders (order_code, gmc_business_id)
  VALUES (p_code, p_business_id)
  RETURNING id;
$$ LANGUAGE sql;
```

```ts
const { data, error } = await supabase.rpc('insert_gmc_order', {
  p_code: code,
  p_business_id: businessId,
})
```

## Also: order_code generation
```ts
// BAD: rely on DB trigger for order_code (trigger may not exist in production)
const { data } = await supabase.from('orders').insert({ amount: 500000 })
// order_code is null — trigger wasn't deployed

// GOOD: generate in RPC or application code
const orderCode = `WZ${Date.now().toString().slice(-8)}`
```

## Why
Real bugs: commits 9b16b85, b3024d1, f3de966. PostgREST caches schema on startup. After migration adds columns, cache is stale until restart. RPC functions execute directly in PostgreSQL — no schema cache involved. Also safer for generated values (order_code) that depend on triggers which may not be deployed.
