# Bug: Banner reorder endpoint thiếu validation array length

## Mô tả
Endpoint `/reorder` nhận array `$ids` từ client nhưng không giới hạn độ dài. Attacker có thể gửi array 10,000+ IDs → N queries `$wpdb->update()` → DoS.

## Tại sao nguy hiểm
- **DoS attack**: 10,000 IDs = 10,000 UPDATE queries
- **Database lock**: Nhiều UPDATE cùng lúc lock table
- **Memory exhaustion**: `array_map('absint', $ids)` với array lớn

## Code sai (BannerController.php:88-98)
```php
public function reorder_banners( WP_REST_Request $request ): WP_REST_Response {
    $data = $request->get_json_params();
    $ids  = $data['ids'] ?? array();

    if ( empty( $ids ) || ! is_array( $ids ) ) {
        return new WP_REST_Response( array( 'error' => 'Danh sách ID là bắt buộc' ), 400 );
    }

    $result = BannerRepository::reorder( array_map( 'absint', $ids ) );
    return new WP_REST_Response( array( 'success' => $result ) );
}
```

**Vấn đề**: Không check `count($ids)` → có thể nhận 10,000+ IDs.

## Code đúng
```php
public function reorder_banners( WP_REST_Request $request ): WP_REST_Response {
    $data = $request->get_json_params();
    $ids  = $data['ids'] ?? array();

    if ( empty( $ids ) || ! is_array( $ids ) ) {
        return new WP_REST_Response( array( 'error' => 'Danh sách ID là bắt buộc' ), 400 );
    }

    // Giới hạn max 100 items để tránh DoS
    if ( count( $ids ) > 100 ) {
        return new WP_REST_Response( array( 'error' => 'Tối đa 100 items' ), 400 );
    }

    $result = BannerRepository::reorder( array_map( 'absint', $ids ) );
    return new WP_REST_Response( array( 'success' => $result ) );
}
```

## Cách fix
1. Thêm validation `count($ids) > 100` trước khi xử lý
2. Giới hạn hợp lý: 100 banners là đủ cho UI admin
3. Return 400 Bad Request nếu vượt limit

## Áp dụng cho
- Mọi endpoint nhận array từ client: reorder, bulk delete, bulk update
- Đặc biệt quan trọng với `manage_options` permission (admin có thể tự DoS)
