## Problem

Public REST endpoints (no auth required) without rate limiting allow attackers to:
- Spam subscriber lists with fake emails (subscribe endpoint)
- Inflate analytics stats (track endpoint)
- DoS the database with rapid-fire requests

## Bad

```php
array(
    'route'      => '/subscribe',
    'method'     => 'POST',
    'callback'   => array( $controller, 'subscribe' ),
    'permission' => 'public', // No rate limit!
),
```

## Good

```php
public function subscribe( \WP_REST_Request $request ): \WP_REST_Response {
    $ip = $_SERVER['REMOTE_ADDR'] ?? '';
    $key = 'wz_popup_rate_' . md5( $ip );
    $count = (int) get_transient( $key );

    if ( $count >= 5 ) {
        return new \WP_REST_Response( array( 'error' => 'Rate limit exceeded' ), 429 );
    }

    set_transient( $key, $count + 1, MINUTE_IN_SECONDS );
    // ... proceed with subscribe logic
}
```

## Apply

- Every public REST endpoint that writes data MUST have rate limiting
- Use WordPress transients with IP-based keys
- Reasonable limits: 5 subscribes/min, 30 track events/min per IP
- Return HTTP 429 when exceeded
