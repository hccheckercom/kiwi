## Bad
```ts
// Supabase select returns untyped data when schema is complex
const { data } = await supabase.from('backlinks').select('*')
// data is typed as any[] in some edge cases → implicit any propagates
// Build error: Parameter 'row' implicitly has an 'any' type
data.map(row => row.url) // TS error in strict mode
```

## Good (typed interface pattern)
```ts
interface BLRow {
  id: string
  url: string
  status: string
  created_at: string
}

const { data } = await supabase.from('backlinks').select('id, url, status, created_at')
const rows = (data ?? []) as BLRow[]
rows.map(row => row.url) // Typed correctly
```

## Good (typed helper function)
```ts
function seoFrom<T>(table: string, select: string) {
  return async (): Promise<T[]> => {
    const supabase = await createClient()
    const { data } = await supabase.from(table).select(select)
    return (data ?? []) as T[]
  }
}

const getBacklinks = seoFrom<BLRow>('backlinks', 'id, url, status')
```

## Why
Real bugs: commits e6d1b35, 42d4407, 755a1b0, 3baafcd. Supabase TypeScript inference breaks on complex schemas (generated columns, views, RPC returns). Build fails with "implicit any" in strict mode. Fix: define explicit interfaces for query results and cast once at the boundary, not throughout the code.
