# Bug: SQL Column Name Does Not Match Schema

## Problem

SQL query references a column that doesn't exist in the table schema. This causes either:
- MySQL error (strict mode): query fails entirely
- MySQL warning (lenient mode): returns NULL silently — data appears missing

Common causes:
- Renaming columns in schema migration but not updating queries
- Copy-pasting queries from another table
- Using WordPress column names (user_id) instead of custom table names (customer_id)

**Example (BAD):**
```php
// Schema: wz_theme_orders has 'customer_id' column
// Query uses 'user_id' which doesn't exist
$rows = $wpdb->get_results(
    "SELECT o.id, o.theme_product_id, o.user_id, o.amount ...
     FROM {$orders_table} o"
);
// In strict SQL mode: ERROR 1054 Unknown column 'o.user_id'
// In lenient mode: returns NULL for user_id
```

## Fix

Always verify column names against the CREATE TABLE schema:

```php
// Schema has: customer_id bigint(20) unsigned NOT NULL DEFAULT 0
$rows = $wpdb->get_results(
    "SELECT o.id, o.theme_product_id, o.customer_id, o.amount ...
     FROM {$orders_table} o"
);
```

## Prevention

- After schema migration: grep all queries for renamed columns
- Use constants or a schema mapping for column names
- Test queries against actual database after migration
