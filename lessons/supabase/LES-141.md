## Bad
```ts
// After adding new column via migration, insert immediately fails
const { error } = await supabase.from('orders').insert({
  order_code: code,
  gmc_business_id: businessId, // NEW column — schema cache doesn't know about it yet
})
// Error: "column gmc_business_id does not exist" (but it does in DB!)
```

## Workaround 1: RPC function
```ts
// Create an RPC function that bypasses PostgREST schema cache
const { error } = await supabase.rpc('insert_gmc_order', {
  p_order_code: code,
  p_business_id: businessId,
})
```

## Workaround 2: Direct REST API
```ts
// Bypass PostgREST entirely — hit Supabase REST directly
const res = await fetch(`${process.env.NEXT_PUBLIC_SUPABASE_URL}/rest/v1/orders`, {
  method: 'POST',
  headers: {
    'apikey': process.env.SUPABASE_SERVICE_ROLE_KEY!,
    'Authorization': `Bearer ${process.env.SUPABASE_SERVICE_ROLE_KEY!}`,
    'Content-Type': 'application/json',
    'Prefer': 'return=representation',
  },
  body: JSON.stringify({ order_code: code, gmc_business_id: businessId }),
})
```

## Fix (proper): Reload PostgREST schema cache
```sql
-- After migration, notify PostgREST to reload
NOTIFY pgrst, 'reload schema';
```

## Why
Real bugs: commits b3024d1, f3de966, 7c94882. Supabase PostgREST caches the DB schema. After running a migration that adds columns, the cache is stale until PostgREST restarts or receives NOTIFY. In production, this causes insert failures for new columns even though the migration succeeded.
