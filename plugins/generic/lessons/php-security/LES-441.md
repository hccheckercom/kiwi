## Problem

Using `{$column}` in SQL queries for column names bypasses `$wpdb->prepare()` protection. Even if the variable is currently set from internal code, a future refactor could pass user input, creating SQL injection.

## Bad

```php
private function upsert_stat( string $popup_type, string $variant, string $column ): void {
    $wpdb->query( $wpdb->prepare(
        "INSERT INTO {$table} (...) VALUES (%s, %s, %d, %d, %s)
        ON DUPLICATE KEY UPDATE {$column} = {$column} + 1", // $column not validated!
        ...
    ));
}
```

## Good

```php
private function upsert_stat( string $popup_type, string $variant, string $column ): void {
    $allowed = array( 'views', 'conversions' );
    if ( ! in_array( $column, $allowed, true ) ) {
        return;
    }

    $wpdb->query( $wpdb->prepare(
        "INSERT INTO {$table} (...) VALUES (%s, %s, %d, %d, %s)
        ON DUPLICATE KEY UPDATE {$column} = {$column} + 1",
        ...
    ));
}
```

## Apply

- Any SQL query using a PHP variable for column/table name MUST validate against an explicit allowlist
- `$wpdb->prepare()` only protects VALUES, not identifiers
- Use `in_array( $column, $allowed, true )` before interpolation
