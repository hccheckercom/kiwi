## Pattern

REST endpoints that accept arrays (e.g., `ids`, `items`, `order`) from request body without enforcing a maximum size are vulnerable to denial-of-service. An attacker can send arrays with thousands of items, causing excessive DB queries, memory usage, or lock contention.

## Bug

```php
// SAI — no size limit
public function reorder_sections( WP_REST_Request $request ): WP_REST_Response {
    $data = $request->get_json_params();
    $ids  = $data['ids'] ?? array();

    if ( empty( $ids ) || ! is_array( $ids ) ) {
        return new WP_REST_Response( array( 'error' => 'Invalid' ), 400 );
    }

    // Attacker sends 10,000 IDs → 10,000 UPDATE queries
    PageSectionRepository::reorder( $ids );
    return new WP_REST_Response( array( 'success' => true ) );
}
```

## Fix

```php
// ĐÚNG — validate size + sanitize values
public function reorder_sections( WP_REST_Request $request ): WP_REST_Response {
    $data = $request->get_json_params();
    $ids  = $data['ids'] ?? array();

    if ( empty( $ids ) || ! is_array( $ids ) ) {
        return new WP_REST_Response( array( 'error' => 'Invalid' ), 400 );
    }

    if ( count( $ids ) > 100 ) {
        return new WP_REST_Response( array( 'error' => 'Too many items (max 100)' ), 400 );
    }

    $ids = array_map( 'absint', $ids );
    $ids = array_filter( $ids );

    PageSectionRepository::reorder( $ids );
    return new WP_REST_Response( array( 'success' => true ) );
}
```

## Áp dụng

1. **Always limit array size** — max 100 for reorder, max 1000 for bulk operations
2. **Sanitize each element** — `absint()` for IDs, `sanitize_key()` for strings
3. **Filter invalid values** — `array_filter()` to remove zeros/empty
4. **Document limits** in API schema
