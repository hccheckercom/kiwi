## Bad
```ts
await supabase.from('orders').update({ status: 'shipped' }).eq('id', orderId)
```
Admin marks order as "shipped" while customer cancels at the same time → last write wins, cancel is lost.

## Good
```ts
const { data, error } = await supabase
  .from('orders')
  .update({ status: 'shipped', updated_at: new Date().toISOString() })
  .eq('id', orderId)
  .eq('status', 'paid') // Only transition from expected state
  .select()

if (!data?.length) throw new Error('Order status changed — please refresh')
```

## Why
Order status is a state machine: pending → paid → shipped → delivered. Without a WHERE clause on current status, concurrent updates skip states or lose transitions. Always assert the expected current state in the UPDATE.
