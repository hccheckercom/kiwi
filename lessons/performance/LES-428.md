## Pattern

A price filter hook (`PRODUCT_PRICE`, `woocommerce_product_get_price`, etc.) runs a DB query
for every product rendered on the page. With 20 products per page, that's 20 identical queries.

## Bad

```php
public function apply_tier_discount( float $price, int $product_id ): float {
    $user_id = get_current_user_id();
    // This runs for EVERY product on the page
    $tier = $wpdb->get_var( $wpdb->prepare(
        "SELECT tier FROM {$table} WHERE user_id = %d", $user_id
    ) );
    $discount = $this->get_tier_discount( $tier );
    return $price * ( 1 - $discount / 100 );
}
```

## Good

```php
public function apply_tier_discount( float $price, int $product_id ): float {
    $user_id  = get_current_user_id();
    $cache_key = "wz_loyalty_tier_{$user_id}";
    $tier = wp_cache_get( $cache_key, 'wezone-loyalty' );
    if ( false === $tier ) {
        $tier = $this->get_user_tier( $user_id );
        wp_cache_set( $cache_key, $tier, 'wezone-loyalty', HOUR_IN_SECONDS );
    }
    $discount = $this->get_tier_discount( $tier );
    if ( $discount <= 0 ) {
        return (float) $price;
    }
    return round( (float) $price * ( 1 - $discount / 100 ), 0 );
}
```

## Why

Each `get_var()` inside a filter that runs per-product creates O(N) DB queries where
N = number of products. On archive/category pages with 20-50 products, this adds
significant latency and DB load.
