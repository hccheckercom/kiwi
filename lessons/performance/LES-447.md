# Bug: wp_schedule_single_event() Race Condition

## Problem

Calling `wp_schedule_single_event(time(), 'hook')` on every page load (when a transient is missing) can schedule duplicate cron events if multiple admin users load the dashboard simultaneously.

WordPress stores cron events by exact timestamp. Two requests at `time()` = 1716537600 will overwrite each other (OK), but if they differ by 1 second, both schedule separate events.

**Example (BAD):**
```php
if (empty($cached)) {
    wp_schedule_single_event(time(), 'wz_score_speed_check');
    // Admin A: schedules at 1716537600
    // Admin B: schedules at 1716537601 (1 second later)
    // → TWO speed checks run
}
```

## Fix

Check if event is already scheduled before adding:

```php
if (empty($cached) && !wp_next_scheduled('wz_score_speed_check')) {
    wp_schedule_single_event(time() + 5, 'wz_score_speed_check');
}
```

Or use a transient-based lock:

```php
if (empty($cached) && false === get_transient('wz_score_speed_lock')) {
    set_transient('wz_score_speed_lock', 1, 60);
    wp_schedule_single_event(time() + 5, 'wz_score_speed_check');
}
```

## Impact

- Prevents duplicate HTTP requests to homepage for speed checks
- Reduces unnecessary cron queue bloat
