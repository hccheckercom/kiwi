## Pattern

A method that writes to a DB column with ENUM constraint accepts an arbitrary string parameter
without validating it against the allowed values. MySQL silently inserts empty string for
invalid ENUM values (in non-strict mode) or throws error (in strict mode).

## Bad

```php
public function log_transaction( int $user_id, string $type, int $points, ... ): bool {
    // $type can be anything — 'earn', 'redeem', 'expire', 'adjust', or 'INVALID'
    $wpdb->insert( $table, array( 'type' => $type, ... ) );
}
```

## Good

```php
private const ALLOWED_TYPES = array( 'earn', 'redeem', 'expire', 'adjust' );

public function log_transaction( int $user_id, string $type, int $points, ... ): bool {
    if ( ! in_array( $type, self::ALLOWED_TYPES, true ) ) {
        wz_log( "Invalid transaction type: {$type}", 'error' );
        return false;
    }
    $wpdb->insert( $table, array( 'type' => $type, ... ) );
}
```

## Why

If MySQL runs in non-strict mode, invalid ENUM values silently become empty strings,
corrupting data and breaking queries that filter by type. In strict mode, the INSERT fails
entirely. Either way, validating at the PHP layer gives clear error messages and prevents
data corruption.
