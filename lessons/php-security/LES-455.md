# Bug: Missing per_page Bounds Check

## Problem

REST endpoints accepting `per_page` parameter without bounds checking allow attackers (even authenticated admin) to request `per_page=999999`, causing:
- Full table scan (no LIMIT optimization)
- Memory exhaustion on large datasets
- Slow response / timeout

**Example (BAD):**
```php
$per_page = (int) $request->get_param( 'per_page' ) ?: 20;
// Attacker sends per_page=999999 → loads entire table
```

## Fix

Always clamp per_page with min/max:

```php
$per_page = min( 100, max( 1, (int) ( $request->get_param( 'per_page' ) ?: 20 ) ) );
```

Or use WP_REST_Server built-in schema:
```php
register_rest_route( 'wezone/v1', '/items', array(
    'args' => array(
        'per_page' => array(
            'type'    => 'integer',
            'minimum' => 1,
            'maximum' => 100,
            'default' => 20,
        ),
    ),
) );
```

## Impact

- Prevents memory exhaustion and slow queries
- Consistent with WordPress REST API conventions (max 100)
