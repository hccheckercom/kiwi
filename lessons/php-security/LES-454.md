# Bug: $_SERVER Without wp_unslash()

## Problem

WordPress adds magic quotes (slashes) to superglobals via `wp_magic_quotes()`. Accessing `$_SERVER` values directly without `wp_unslash()` can leave backslashes in data like IP addresses, user agents, and headers.

PHPCS WPCS rule: `WordPress.Security.ValidatedSanitizedInput`

**Example (BAD):**
```php
$ip = $_SERVER['HTTP_CF_CONNECTING_IP']
    ?? $_SERVER['HTTP_X_FORWARDED_FOR']
    ?? $_SERVER['REMOTE_ADDR']
    ?? '';
// May contain slashes from wp_magic_quotes()
```

## Fix

```php
$ip = sanitize_text_field( wp_unslash(
    $_SERVER['HTTP_CF_CONNECTING_IP']
    ?? $_SERVER['HTTP_X_FORWARDED_FOR']
    ?? $_SERVER['REMOTE_ADDR']
    ?? ''
) );
```

Or per-key:
```php
$headers = array( 'HTTP_CF_CONNECTING_IP', 'HTTP_X_FORWARDED_FOR', 'REMOTE_ADDR' );
foreach ( $headers as $header ) {
    if ( ! empty( $_SERVER[ $header ] ) ) {
        $ip = sanitize_text_field( wp_unslash( $_SERVER[ $header ] ) );
        break;
    }
}
```

## Impact

- WPCS lint failure
- Potential data corruption if slashes in IP or user-agent strings
