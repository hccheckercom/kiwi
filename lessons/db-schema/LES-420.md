# Bug Pattern

Updating `sold_count` or similar counters with `SET sold_count = sold_count + N` in separate queries creates race conditions during concurrent checkouts.

**Why:** Two simultaneous checkouts can read the same `sold_count`, both increment it, and both write back → one sale is lost from the counter.

## Detection

```regex
wpdb->query\([^)]*UPDATE[^)]*SET\s+\w*(?:count|sold|qty)\w*\s*=\s*\w*(?:count|sold|qty)\w*\s*\+
```

Match UPDATE queries that increment counters without atomic operations.

## Examples

### ❌ BAD: Non-atomic counter update

```php
public function record_sale( int $order_id ): void {
    global $wpdb;
    $table = $wpdb->prefix . 'wezone_flash_sale_products';
    
    foreach ( wz_get_order_items( $order_id ) as $item ) {
        // RACE CONDITION: Two concurrent checkouts can both read sold_count=10,
        // both increment to 11, both write 11 → one sale lost
        $wpdb->query(
            $wpdb->prepare(
                "UPDATE {$table} SET sold_count = sold_count + %d WHERE sale_id = %d",
                (int) $item['qty'],
                $sale['sale_id']
            )
        );
    }
}
```

**Timeline of race condition:**
```
Time | Checkout A (qty=2)           | Checkout B (qty=3)           | DB sold_count
-----|------------------------------|------------------------------|---------------
T0   | Read sold_count=10           |                              | 10
T1   |                              | Read sold_count=10           | 10
T2   | Write sold_count=10+2=12     |                              | 12
T3   |                              | Write sold_count=10+3=13     | 13 (should be 15!)
```

### ✅ GOOD: Atomic increment with WHERE clause

```php
public function record_sale( int $order_id ): void {
    global $wpdb;
    $table = $wpdb->prefix . 'wezone_flash_sale_products';
    
    foreach ( wz_get_order_items( $order_id ) as $item ) {
        // Atomic: MySQL evaluates sold_count + N at write time, not read time
        $affected = $wpdb->query(
            $wpdb->prepare(
                "UPDATE {$table} 
                 SET sold_count = sold_count + %d 
                 WHERE sale_id = %d 
                   AND product_id = %d
                   AND (stock_limit IS NULL OR sold_count + %d <= stock_limit)",
                (int) $item['qty'],
                $sale['sale_id'],
                $item['product_id'],
                (int) $item['qty']
            )
        );
        
        // Verify update succeeded (stock_limit check might have failed)
        if ( $affected === 0 ) {
            wz_log( "FlashSale: stock_limit exceeded for product {$item['product_id']}", 'error' );
            // Optionally: cancel order or remove item from order
        }
    }
}
```

### ✅ BETTER: Use database transaction for multi-item orders

```php
public function record_sale( int $order_id ): void {
    global $wpdb;
    $table = $wpdb->prefix . 'wezone_flash_sale_products';
    
    // Start transaction to ensure all-or-nothing update
    $wpdb->query( 'START TRANSACTION' );
    
    $all_success = true;
    
    foreach ( wz_get_order_items( $order_id ) as $item ) {
        $affected = $wpdb->query(
            $wpdb->prepare(
                "UPDATE {$table} 
                 SET sold_count = sold_count + %d 
                 WHERE sale_id = %d 
                   AND product_id = %d
                   AND (stock_limit IS NULL OR sold_count + %d <= stock_limit)",
                (int) $item['qty'],
                $sale['sale_id'],
                $item['product_id'],
                (int) $item['qty']
            )
        );
        
        if ( $affected === 0 ) {
            $all_success = false;
            wz_log( "FlashSale: stock_limit exceeded for product {$item['product_id']}", 'error' );
            break;
        }
    }
    
    if ( $all_success ) {
        $wpdb->query( 'COMMIT' );
    } else {
        $wpdb->query( 'ROLLBACK' );
        // Optionally: mark order as failed or pending review
    }
}
```

## Fix Strategy

1. **Always use atomic SQL**: `SET count = count + N` in single query
2. **Add WHERE clause**: Include stock_limit check in UPDATE to prevent overselling
3. **Check affected rows**: Verify UPDATE succeeded (0 rows = limit exceeded)
4. **Use transactions**: For multi-item updates, wrap in START TRANSACTION / COMMIT
5. **Log failures**: Record when stock_limit prevents update
