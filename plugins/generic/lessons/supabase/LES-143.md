## Bad
```ts
// Two webhooks arrive 50ms apart for same order
const { data: order } = await supabase.from('orders').select('*').eq('order_code', code).single()
if (order.status === 'pending') {
  // Both webhooks pass this check simultaneously!
  await supabase.from('orders').update({ status: 'paid' }).eq('id', order.id)
  await sendEmail(order) // Email sent twice
}
```

## Good (optimistic lock pattern from webhook.handler.ts)
```ts
const { data: order } = await supabase.from('orders')
  .select('id, status').eq('order_code', code).single()

if (order.status !== 'pending') return Response.json({ success: true }) // Idempotent

// Optimistic lock: WHERE status = 'pending' ensures only ONE webhook wins
const { data: updated } = await supabase.from('orders')
  .update({ status: 'paid', payment_ref: refCode })
  .eq('id', order.id)
  .eq('status', 'pending')  // ← This is the lock
  .select('id')

// Only proceed if WE were the one who updated (not the other webhook)
if (!updated?.length) return Response.json({ success: true })

await sendEmail(order) // Only fires once
```

## Why
Real pattern from webhook.handler.ts. Payment gateways (SePay, VNPay) retry webhooks. Without optimistic lock (.eq('status', 'pending') in the UPDATE WHERE clause), concurrent webhooks both read status='pending', both update, both send emails. The WHERE clause makes the UPDATE atomic — only one succeeds.
