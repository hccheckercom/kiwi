## Pattern

Order status transition reads current status then updates in separate queries without row-level locking. Two concurrent requests (e.g., webhook + admin) can both read `pending`, both pass `canTransition()`, and both try to transition — leading to invalid state or duplicate hooks.

## Wrong

```php
// Thread A reads status = 'pending', Thread B reads status = 'pending'
$order = $wpdb->get_row(
    $wpdb->prepare("SELECT status FROM {$wpdb->prefix}wz_orders WHERE id = %d", $order_id),
    ARRAY_A
);
// Both pass canTransition('pending', 'processing') → both update
$wpdb->update($wpdb->prefix . 'wz_orders', ['status' => 'processing'], ['id' => $order_id]);
```

## Correct

```php
$wpdb->query('START TRANSACTION');
$order = $wpdb->get_row(
    $wpdb->prepare("SELECT status FROM {$wpdb->prefix}wz_orders WHERE id = %d FOR UPDATE", $order_id),
    ARRAY_A
);
if (!$order || !$this->canTransition($order['status'], $to)) {
    $wpdb->query('ROLLBACK');
    return false;
}
$wpdb->update($wpdb->prefix . 'wz_orders', ['status' => $to, 'updated_at' => gmdate('Y-m-d H:i:s')], ['id' => $order_id]);
$wpdb->query('COMMIT');
```

## Why

Payment webhooks and admin actions can fire simultaneously. Without `FOR UPDATE`, the race window is ~50ms — enough for duplicate payment processing, double stock deduction, or double email sending.
