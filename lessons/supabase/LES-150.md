## Bad
```ts
// Assumes 'stores' table exists in webstore-vn Supabase — it doesn't!
const { data } = await supabase.from('stores').select('*')
// Error: relation "public.stores" does not exist
```

## Table mapping in webstore-vn
| What you want | Correct table | Wrong table |
|---------------|---------------|-------------|
| Demo store list (demo.wezone.vn) | External API: `/api/demo-stores` proxy | `stores` (doesn't exist) |
| Template catalog | `templates` | `stores` |
| Platform multi-store | `platform_stores` | `stores` |
| Demo store data | `demo_stores` (on demo.wezone.vn DB) | `stores` on main DB |

## Good
```ts
// For demo showcase on homepage — use proxy API (avoids CORS + correct DB)
const res = await fetch('/api/demo-stores')
const { data } = await res.json()

// For template catalog
const { data } = await supabase.from('templates').select('*')

// For platform stores (admin)
const { data } = await supabase.from('platform_stores').select('*').eq('owner_id', userId)
```

## Why
Real bugs: commits c5e1a78, 0473d11, 29526c5. webstore-vn has multiple "store" concepts across different databases. Querying the wrong table returns empty results or errors. The demo stores live on demo.wezone.vn's separate Supabase instance — must use proxy API route, not direct query.
