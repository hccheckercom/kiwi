## Pattern

`OrderStateMachine::transition()` updates the order status then inserts a history record in separate, non-transactional operations. If the history insert fails, the order is in a new status but the audit trail is broken.

## Wrong

```php
$wpdb->update($wpdb->prefix . 'wz_orders', ['status' => $to], ['id' => $order_id]);
// ← no transaction boundary
$history->record($order_id, $from, $to, $note, $by);
```

## Correct

```php
$wpdb->query('START TRANSACTION');
$updated = $wpdb->update($wpdb->prefix . 'wz_orders', ['status' => $to], ['id' => $order_id]);
if ($updated === false) {
    $wpdb->query('ROLLBACK');
    return false;
}
$hist_result = $history->record($order_id, $from, $to, $note, $by);
if ($hist_result === 0) {
    $wpdb->query('ROLLBACK');
    return false;
}
$wpdb->query('COMMIT');
```

## Why

Order status without matching history record breaks admin audit trail, customer notification logs, and status change debugging. Both operations must succeed or both must fail.
