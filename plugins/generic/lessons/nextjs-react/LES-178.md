## Bad
```ts
const schema = z.object({
  price: z.string().transform(v => Number(v)),  // NaN if empty string
  quantity: z.string().transform(v => parseInt(v, 10)), // NaN if "abc"
})

const result = schema.parse(Object.fromEntries(formData))
// result.price = NaN → silently corrupts DB
```

## Good (Zod v4 z.coerce)
```ts
const schema = z.object({
  price: z.coerce.number().min(0),      // Handles string→number, rejects NaN
  quantity: z.coerce.number().int().min(1),
  email: z.string().email(),            // String fields stay as z.string()
  is_active: z.coerce.boolean(),        // "true"/"false" strings → boolean
})

const result = schema.safeParse(Object.fromEntries(formData))
if (!result.success) return { error: result.error.flatten().fieldErrors }
```

## z.coerce behavior
| Input | z.coerce.number() | z.string().transform(Number) |
|-------|-------------------|------------------------------|
| "123" | 123 | 123 |
| "" | Error (NaN rejected) | NaN (passes!) |
| "abc" | Error (NaN rejected) | NaN (passes!) |
| null | Error | TypeError |

## Why
FormData values are always strings. z.coerce.number() uses Number() internally but then validates the result — NaN fails validation. Manual .transform(Number) passes NaN through as a valid number. In webstore-vn, this caused 0-price products and NaN quantities in platform_products table.
