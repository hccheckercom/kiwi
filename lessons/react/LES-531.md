## Bad
```tsx
async function loadGraph() {
  const response = await axios.post('/api/plan', {...})
  const { dependency_graph } = response.data
  setLoading(false)
  renderGraph(dependency_graph.nodes, dependency_graph.links) // Called immediately after setState
}

function renderGraph(nodes, links) {
  if (!svgRef.current) { // svgRef.current is null - React hasn't re-rendered yet
    console.error('svgRef.current is null!')
    return
  }
  // D3 rendering code...
}
```

## Good
```tsx
const [graphData, setGraphData] = useState(null)

async function loadGraph() {
  const response = await axios.post('/api/plan', {...})
  const { dependency_graph } = response.data
  setGraphData(dependency_graph) // Store in state
  setLoading(false)
  // Don't call renderGraph here
}

// Separate useEffect watches for both data AND ref availability
useEffect(() => {
  if (!loading && graphData && svgRef.current) {
    renderGraph(graphData.nodes, graphData.links)
  }
}, [loading, graphData]) // Runs after React re-renders
```

## Why
Calling DOM manipulation functions immediately after setState causes race condition. React batches state updates and re-renders asynchronously. When you call `setLoading(false)` then immediately call `renderGraph()`, the component hasn't re-rendered yet, so `svgRef.current` is still null. Solution: store data in state and use separate useEffect that watches for BOTH data availability AND ref existence.
