# Bug: get_post_meta() N+1 Queries In Loop

## Problem

Calling `get_post_meta()` inside a `foreach` loop causes N queries per meta key per iteration.
WordPress caches meta per-object after first fetch, but only if `update_meta_cache()` was called or the post was loaded via `WP_Query` (which primes the cache automatically).

When products come from a custom function like `wz_get_products()` that returns raw arrays (not WP_Post objects loaded via WP_Query), the meta cache is NOT primed — every `get_post_meta()` call hits the DB.

**Example (BAD):**
```php
$products = wz_get_products(['status' => 'publish', 'limit' => 200]);
foreach ($products as $product) {
    $title = get_post_meta($product['id'], '_wezone_seo_title', true);   // query 1
    $desc  = get_post_meta($product['id'], '_wezone_seo_description', true); // query 2
}
// 200 products × 2 meta keys = 400 queries
```

## Fix

Prime the meta cache before the loop with a single query:

```php
$products = wz_get_products(['status' => 'publish', 'limit' => 200]);
$ids = wp_list_pluck($products, 'id');

// 1 query primes ALL meta for all IDs
update_meta_cache('post', $ids);

foreach ($products as $product) {
    // Now these hit the in-memory cache — 0 queries
    $title = get_post_meta($product['id'], '_wezone_seo_title', true);
    $desc  = get_post_meta($product['id'], '_wezone_seo_description', true);
}
```

## Impact

- 200 products × 2 meta keys = **400 queries → 1 query**
- Dashboard widget loads 2-5 seconds faster on stores with many products
