## Bad
```ts
export const GET = withPlatformKey(async (req, store) => {
  const supabase = await createRawAdminClient()
  // BUG: No store_id filter — returns ALL stores' orders!
  const { data } = await supabase.from('platform_orders').select('*')
  return apiSuccess(data)
})
```

## Good
```ts
export const GET = withPlatformKey(async (req, store) => {
  const supabase = await createRawAdminClient()
  // Always scope to current store — tenant isolation
  const { data } = await supabase
    .from('platform_orders')
    .select('*')
    .eq('store_id', store.id)
    .order('created_at', { ascending: false })
    .range(0, 19)
  return apiSuccess(data)
})
```

## Checklist for platform API routes
```
✓ .eq('store_id', store.id) on every query
✓ Insert includes store_id: store.id
✓ Update/Delete has .eq('store_id', store.id) (prevent cross-tenant mutation)
✓ Search params sanitized: .slice(0, 100).replace(/[%_\\]/g, '')
```

## Why
webstore-vn is multi-tenant — platform_* tables contain data from ALL stores. withPlatformKey injects the authenticated store, but you must manually filter by store_id. Missing this filter = one store can see/modify another store's data. This is the #1 security risk in multi-tenant SaaS.
