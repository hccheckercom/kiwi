## Problem

`aggregate_daily_stats()` checks for existing rows before insert, but `aggregate_product_stats()` does not. If the cron job runs twice (overlap, retry), product stats rows are duplicated, inflating product revenue/quantity reports.

## Bad

```php
public function aggregate_product_stats( string $date ): void {
    // No check for existing rows!
    $product_rows = $wpdb->get_results( ... );
    wz_bulk_insert( $products_table, $insert_rows, ... );
}
```

## Good

```php
public function aggregate_product_stats( string $date ): void {
    $existing = $wpdb->get_var(
        $wpdb->prepare( "SELECT COUNT(*) FROM {$products_table} WHERE date = %s", $date )
    );
    if ( $existing ) {
        return;
    }
    $product_rows = $wpdb->get_results( ... );
    wz_bulk_insert( $products_table, $insert_rows, ... );
}
```

## Why

Cron overlap is common under load. Without idempotency guards, duplicate aggregate rows silently corrupt reporting data. The daily table has this guard but the products table was missed.
