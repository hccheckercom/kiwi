# Bug: Banner save_effects không validate hex color format đầy đủ

## Mô tả
`save_effects()` dùng `sanitize_hex_color()` cho `overlay_color` nhưng fallback về `#000000` nếu invalid. Tốt hơn là reject request với 400 Bad Request.

## Tại sao quan trọng
- **UX**: User không biết color bị reject → UI hiển thị #000000 thay vì báo lỗi
- **Data integrity**: Lưu fallback value thay vì giá trị user muốn
- **Silent failure**: Admin không biết input sai

## Code hiện tại (BannerController.php:184-186)
```php
if ( isset( $data['overlay_color'] ) ) {
    $effects['overlay_color'] = sanitize_hex_color( $data['overlay_color'] ) ?: '#000000';
}
```

**Vấn đề**: Silent fallback → user không biết color bị reject.

## Code đúng (Option 1: Reject invalid)
```php
if ( isset( $data['overlay_color'] ) ) {
    $color = sanitize_hex_color( $data['overlay_color'] );
    if ( ! $color ) {
        return new WP_REST_Response( 
            array( 'error' => 'overlay_color phải là hex color hợp lệ (#RRGGBB)' ), 
            400 
        );
    }
    $effects['overlay_color'] = $color;
}
```

## Code đúng (Option 2: Validate trước sanitize)
```php
if ( isset( $data['overlay_color'] ) ) {
    $color = $data['overlay_color'];
    
    // Validate format first
    if ( ! preg_match( '/^#[0-9A-Fa-f]{6}$/', $color ) ) {
        return new WP_REST_Response( 
            array( 'error' => 'overlay_color phải là hex color 6 ký tự (#RRGGBB)' ), 
            400 
        );
    }
    
    $effects['overlay_color'] = sanitize_hex_color( $color );
}
```

## Cách fix
1. **Preferred**: Validate format trước, reject nếu invalid
2. Return 400 Bad Request với error message rõ ràng
3. Không dùng silent fallback cho user input
4. Fallback chỉ dùng cho default values khi field không được gửi

## Áp dụng cho
- Mọi color input từ admin UI
- Enum validation (status, type, layout)
- Numeric range validation (price, quantity)
