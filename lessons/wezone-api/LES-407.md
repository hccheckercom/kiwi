# Repository cache invalidation pattern

## Problem
Repository methods that modify data (update/delete) must invalidate related caches. Missing cache invalidation causes stale data to be served from cache after updates.

## Pattern
Before update/delete operations:
1. Fetch current row to get cache keys (slug, SKU, parent_id)
2. Perform the operation
3. Delete all related cache entries using fetched keys

## Bad Examples

```php
// BAD: Update without cache invalidation
public function update( int $id, array $data ): bool {
    global $wpdb;
    $result = $wpdb->update( 
        $wpdb->prefix . 'wz_products', 
        $data, 
        array( 'id' => $id ) 
    );
    wp_cache_delete( "wz_product_{$id}", 'wezone_products' );
    return $result !== false;
}
```

**Issue**: Only deletes ID-based cache, but product may also be cached by slug/SKU.

```php
// BAD: Delete without fetching current row
public function delete( int $id ): bool {
    global $wpdb;
    $result = $wpdb->delete( $wpdb->prefix . 'wz_variations', array( 'id' => $id ) );
    return (bool) $result;
}
```

**Issue**: Variation's parent product cache (`wz_variations_{product_id}`) is not invalidated.

## Good Examples

```php
// GOOD: Fetch before update to bust all caches
public function update( int $id, array $data ): bool {
    global $wpdb;
    
    // Fetch current row BEFORE update
    $current = $this->get( $id );
    
    $result = $wpdb->update( 
        $wpdb->prefix . 'wz_products', 
        $data, 
        array( 'id' => $id ) 
    );
    
    if ( $result === false ) {
        return false;
    }
    
    // Invalidate all related caches
    wp_cache_delete( "wz_product_{$id}", 'wezone_products' );
    if ( $current ) {
        if ( ! empty( $current['slug'] ) ) {
            wp_cache_delete( 'wz_product_slug_' . md5( $current['slug'] ), 'wezone_products' );
        }
        if ( ! empty( $current['sku'] ) ) {
            wp_cache_delete( 'wz_product_sku_' . md5( $current['sku'] ), 'wezone_products' );
        }
    }
    
    return true;
}
```

```php
// GOOD: Fetch before delete to bust parent cache
public function delete( int $id ): bool {
    global $wpdb;
    
    // Fetch current row BEFORE delete
    $existing = $this->get( $id );
    
    $result = (bool) $wpdb->delete( $wpdb->prefix . 'wz_variations', array( 'id' => $id ) );
    
    if ( $result && $existing ) {
        $product_id = (int) ( $existing['product_id'] ?? 0 );
        if ( $product_id > 0 ) {
            wp_cache_delete( "wz_variations_{$product_id}", 'wezone_products' );
        }
    }
    
    return $result;
}
```

## Detection Rules
- Repository `update()` method without fetching current row before operation
- Repository `delete()` method without fetching current row before operation
- Cache invalidation only by ID when entity has slug/SKU/parent_id fields
- Missing `wp_cache_delete()` for related entities (variations → parent product)

## Why This Matters
- **Data consistency**: Users see stale data after updates
- **Business logic errors**: Decisions based on cached stale data
- **Hard to debug**: Cache issues are intermittent (TTL-dependent)
