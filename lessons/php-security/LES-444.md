## Bad
```php
setcookie( 'wezone_recently_viewed', $value, time() + 30 * DAY_IN_SECONDS, COOKIEPATH, COOKIE_DOMAIN, is_ssl(), false );
// httpOnly=false → any XSS can read this cookie via document.cookie
```

## Good (option A: httpOnly + REST endpoint)
```php
setcookie( 'wezone_recently_viewed', $value, time() + 30 * DAY_IN_SECONDS, COOKIEPATH, COOKIE_DOMAIN, is_ssl(), true );
// Read viewed products via REST API instead of document.cookie
```

## Good (option B: acceptable if JS must read)
```php
// If JS legitimately needs to read the cookie (e.g., for client-side rendering),
// document that this is intentional and ensure no sensitive data is stored
setcookie( 'wezone_recently_viewed', $value, time() + 30 * DAY_IN_SECONDS, COOKIEPATH, COOKIE_DOMAIN, is_ssl(), false );
// INTENTIONAL: JS reads this for client-side recently-viewed display. Contains only product IDs (non-sensitive).
```

## Why
Cookies with `httpOnly=false` are readable by JavaScript. If an XSS vulnerability exists anywhere on the site, the attacker can read all non-httpOnly cookies. For cookies containing only non-sensitive data (product IDs), this is acceptable with documentation. For cookies with session tokens or user data, always use `httpOnly=true`.
