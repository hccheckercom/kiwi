## Three Supabase clients in webstore-vn

| Client | Import from | Use when |
|--------|-------------|----------|
| `createClient()` | `@/lib/supabase/server` | Server Component/Action needing user session (respects RLS) |
| `createAdminClient()` | `@/lib/supabase/server` | Typed admin ops — bypasses RLS, no session needed |
| `createRawAdminClient()` | `@/lib/supabase/server` | Untyped admin — when TS inference breaks on insert/update |
| `createClient()` | `@/lib/supabase/client` | Browser only — 'use client' components |

## Bad
```ts
// API route that needs to read ALL orders (admin view)
import { createClient } from '@/lib/supabase/server'

export async function GET() {
  const supabase = await createClient()
  // BUG: createClient uses anon key + user session → RLS blocks admin from seeing other users' orders
  const { data } = await supabase.from('orders').select('*')
}
```

## Good
```ts
import { createRawAdminClient } from '@/lib/supabase/server'

export async function GET() {
  // Admin route (already auth-gated by middleware) — needs service role to see all data
  const supabase = await createRawAdminClient()
  const { data } = await supabase.from('orders').select('*')
}
```

## When to use createRawAdminClient (untyped)
```ts
// Supabase TS types break on insert when schema has generated columns or complex defaults
// Error: "Type 'never' is not assignable to type..."
const supabase = await createRawAdminClient() // eslint-disable-next-line — untyped intentional
await supabase.from('platform_orders').insert({ ... }) // Works without type errors
```

## Why
Real pattern from webhook.handler.ts, orders.handlers.ts. Using wrong client = silent data access failures. createClient() respects RLS (good for user-facing), createAdminClient() bypasses RLS (needed for cross-user admin ops). createRawAdminClient() exists because Supabase TS inference breaks on complex schemas.
