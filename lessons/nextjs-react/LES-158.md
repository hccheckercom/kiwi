## Bad
```ts
window.addEventListener('message', (event) => {
  // No origin check — any iframe/window can send messages
  const { type, colors } = event.data
  if (type === 'WZ_APPLY_COLORS') {
    applyColors(colors) // Attacker iframe injects malicious CSS values
  }
})
```

## Good (pattern from lib/color/security.ts)
```ts
import { getSecurityConfig, isValidColorScheme } from '@/lib/color/security'

window.addEventListener('message', (event) => {
  // 1. Origin check — only accept from trusted sources
  const { trustedOrigins } = getSecurityConfig()
  if (!trustedOrigins.has(event.origin)) return

  // 2. Schema validation — reject malformed payloads
  if (!isValidColorScheme(event.data)) return

  // 3. Value validation — each color matches strict regex
  applyColors(event.data.colors)
})
```

## Trusted origins detection (multi-context)
```ts
// Works in 3 contexts:
// 1. Next.js: process.env.NEXT_PUBLIC_APP_ORIGIN
// 2. Standalone HTML: <meta name="wz-allowed-origin">
// 3. Fallback: window.location.origin + localhost dev
```

## Why
webstore-vn uses postMessage for color theme preview (parent frame → demo iframe). Without origin validation, any page that embeds the demo iframe can inject arbitrary color values. Combined with CSS injection (values like `red; } * { display:none }`), this becomes a defacement vector. The security layer validates: origin, schema shape, color format regex, max length, and allowed keys.
