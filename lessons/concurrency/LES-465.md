# Bad Code

```php
public function process_ipn( array $params ): void {
    if ( ! $this->verify_ipn( $params ) ) {
        return;
    }
    
    $order_id = (int) $params['order_id'];
    
    // ❌ HIGH: Không check order status trước khi complete
    // → Duplicate IPN (gateway retry) sẽ trigger double inventory deduction
    $this->completePayment( $order_id, $params['txn_id'] ?? '' );
    $this->fire_event( 'payment_completed', $order_id, $params );
}
```

**Race condition scenario:**
1. IPN #1 arrives → `completePayment()` → inventory -1 → status = `processing`
2. IPN #2 arrives (gateway retry, network duplicate) → `completePayment()` again → inventory -1 again
3. Result: Inventory deducted 2x, customer chỉ mua 1 sản phẩm

# Good Code

```php
public function process_ipn( array $params ): void {
    if ( ! $this->verify_ipn( $params ) ) {
        $this->log( 'IPN rejected: invalid signature', 'error' );
        return;
    }
    
    $order_id = (int) $params['order_id'];
    $order    = wz_get_order( $order_id );
    
    if ( ! $order ) {
        $this->log( sprintf( 'IPN: Order #%d not found', $order_id ), 'error' );
        return;
    }
    
    // ✅ MUST: Check order status — idempotency guard
    if ( in_array( $order['status'] ?? '', array( 'processing', 'completed' ), true ) ) {
        $this->log( sprintf( 'IPN: Order #%d already paid (status: %s) — ignoring duplicate', $order_id, $order['status'] ?? '' ) );
        
        // ✅ Log duplicate attempt for fraud detection
        if ( function_exists( 'wz_log' ) ) {
            wz_log( sprintf( 'Duplicate IPN for order #%d — txn: %s', $order_id, $params['txn_id'] ?? '' ), 'warning', 'payment' );
        }
        return;
    }
    
    $this->completePayment( $order_id, $params['txn_id'] ?? '' );
    $this->addOrderNote( $order_id, sprintf( 'Payment confirmed via IPN. Txn: %s', $params['txn_id'] ?? '' ) );
    $this->fire_event( 'payment_completed', $order_id, $params );
}
```

# Why

**Impact:**
- **Inventory corruption:** Kho bị trừ 2x, overselling prevention fails
- **Double fulfillment:** Nếu hook trigger email/shipping label → customer nhận 2 email, 2 label
- **Accounting mismatch:** 1 payment nhưng 2 order completion events

**Root causes:**
1. **Gateway retries:** VNPay/MoMo retry IPN nếu không nhận 200 OK trong 5s
2. **Network duplicates:** Load balancer, proxy có thể duplicate request
3. **No idempotency key:** Không có mechanism để detect duplicate IPN

**Real-world frequency:**
- VNPay: ~2-5% orders nhận duplicate IPN (gateway retry)
- MoMo: ~1% (ít hơn vì timeout dài hơn)
- SePay webhook: ~0.5% (webhook infrastructure ổn định hơn)

**Why not use DB transaction lock?**
- `wz_get_order()` không lock row → race window vẫn tồn tại
- Proper fix: Check status BEFORE calling `completePayment()`, không rely on DB lock
