## Problem

When `$wpdb->insert()` fails (table locked, disk full, constraint violation), returning `false`, the code continues to `delete_transient()` — permanently losing the counter data that was supposed to be persisted to the aggregate table.

## Bad

```php
$wpdb->insert( $daily_table, $data );
// If insert failed, transients are deleted anyway = DATA LOSS
delete_transient( "wz_analytics_views_{$date}" );
delete_transient( "wz_analytics_atc_{$date}" );
```

## Good

```php
$inserted = $wpdb->insert( $daily_table, $data );
if ( false === $inserted ) {
    error_log( "[Analytics] Failed to insert daily stats for {$date}" );
    return; // Keep transients for retry
}
delete_transient( "wz_analytics_views_{$date}" );
delete_transient( "wz_analytics_atc_{$date}" );
```

## Why

Transient counters are the only source of view/cart/checkout data. Once deleted, the data is unrecoverable. The insert must succeed before cleanup.
