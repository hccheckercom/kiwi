## Pattern

A method checks whether a row exists with SELECT, then conditionally does INSERT or UPDATE.
Without `FOR UPDATE` lock or `INSERT ... ON DUPLICATE KEY UPDATE`, concurrent requests
can both see "not exists" and both INSERT, causing duplicate rows or constraint violations.

## Bad

```php
public function update_balance( int $user_id, ... ): void {
    $existing = $wpdb->get_var(
        $wpdb->prepare( "SELECT user_id FROM {$table} WHERE user_id = %d", $user_id )
    );
    if ( $existing ) {
        $wpdb->update( $table, $data, array( 'user_id' => $user_id ) );
    } else {
        $wpdb->insert( $table, $data );
    }
}
```

## Good

```php
public function update_balance( int $user_id, ... ): void {
    // Use INSERT ... ON DUPLICATE KEY UPDATE — atomic, no race condition
    $wpdb->query( $wpdb->prepare(
        "INSERT INTO {$table} (user_id, balance, tier, total_earned, total_redeemed, total_expired, updated_at)
         VALUES (%d, %d, %s, %d, %d, %d, %s)
         ON DUPLICATE KEY UPDATE
         balance = VALUES(balance), tier = VALUES(tier),
         total_earned = VALUES(total_earned), total_redeemed = VALUES(total_redeemed),
         total_expired = VALUES(total_expired), updated_at = VALUES(updated_at)",
        $user_id, $balance, $tier, $earned, $redeemed, $expired, gmdate('Y-m-d H:i:s')
    ) );
}
```

## Why

Two concurrent requests can both SELECT "not exists", then both INSERT, creating
duplicate rows. With `user_id` as PRIMARY KEY this causes a DB error. With
`ON DUPLICATE KEY UPDATE`, the operation is atomic — no race condition possible.
