## Problem

UPDATE-then-INSERT pattern has a TOCTOU race condition. Two concurrent requests can both see `$updated = 0` (row doesn't exist) and both call `set_transient($key, 1)` — the second overwrites the first, losing one count.

## Bad

```php
$updated = $wpdb->query(
    $wpdb->prepare( "UPDATE {$wpdb->options} SET option_value = option_value + 1 WHERE option_name = %s", $option )
);
if ( ! $updated ) {
    set_transient( $key, 1, 2 * DAY_IN_SECONDS );
}
```

## Good

```php
$wpdb->query(
    $wpdb->prepare(
        "INSERT INTO {$wpdb->options} (option_name, option_value, autoload) VALUES (%s, 1, 'no')
         ON DUPLICATE KEY UPDATE option_value = option_value + 1",
        $option
    )
);
```

## Why

`INSERT ... ON DUPLICATE KEY UPDATE` is atomic at the MySQL level — no gap for a second request to interfere. This is the standard pattern for counters in WordPress.
