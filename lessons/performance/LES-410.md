# Repository query optimization — full-scan detection

## Problem
Repository methods that load entire tables without WHERE clause cause performance issues on large datasets. Full table scans should be intentional and documented.

## Pattern
1. **Avoid full scans** when possible — add WHERE clause or LIMIT
2. **If full scan is necessary** (e.g., category tree for menu):
   - Add phpcs:ignore comment explaining why
   - Consider caching the result
   - Document the use case in comment

## Bad Examples

```php
// BAD: Undocumented full table scan
public function tree(): array {
    global $wpdb;
    $rows = $wpdb->get_results(
        "SELECT * FROM {$wpdb->prefix}wz_categories ORDER BY menu_order ASC",
        ARRAY_A
    );
    return $rows ?: array();
}
```

**Issue**: Loads all categories without explanation. Is this intentional?

```php
// BAD: Full scan when parent_id filter would work
public function roots(): array {
    global $wpdb;
    $rows = $wpdb->get_results(
        "SELECT * FROM {$wpdb->prefix}wz_categories ORDER BY menu_order ASC",
        ARRAY_A
    );
    return array_filter( $rows, fn( $r ) => $r['parent_id'] === null );
}
```

**Issue**: Loads all categories then filters in PHP. Should filter in SQL.

## Good Examples

```php
// GOOD: Documented full scan with use case
public function tree( ?int $parent_id = null ): array {
    global $wpdb;
    
    if ( $parent_id === null ) {
        // phpcs:ignore WordPress.DB.PreparedSQL.NotPrepared,WordPress.DB.PreparedSQL.InterpolatedNotPrepared
        $rows = $wpdb->get_results( // full-scan: loads entire category tree for menu rendering
            "SELECT * FROM {$wpdb->prefix}wz_categories ORDER BY menu_order ASC, name ASC",
            ARRAY_A
        );
    } else {
        $rows = $wpdb->get_results(
            $wpdb->prepare(
                "SELECT * FROM {$wpdb->prefix}wz_categories WHERE parent_id = %d ORDER BY menu_order ASC, name ASC",
                $parent_id
            ),
            ARRAY_A
        );
    }
    
    return $rows ?: array();
}
```

```php
// GOOD: Filter in SQL, not PHP
public function roots(): array {
    global $wpdb;
    
    // phpcs:ignore WordPress.DB.PreparedSQL.NotPrepared,WordPress.DB.PreparedSQL.InterpolatedNotPrepared
    return $wpdb->get_results(
        "SELECT * FROM {$wpdb->prefix}wz_categories WHERE parent_id IS NULL ORDER BY menu_order ASC",
        ARRAY_A
    ) ?: array();
}
```

```php
// GOOD: Add LIMIT when full result not needed
public function recent( int $limit = 10 ): array {
    global $wpdb;
    
    return $wpdb->get_results(
        $wpdb->prepare(
            "SELECT * FROM {$wpdb->prefix}wz_products WHERE status = 'publish' ORDER BY created_at DESC LIMIT %d",
            $limit
        ),
        ARRAY_A
    ) ?: array();
}
```

## Detection Rules
- `SELECT * FROM table` without WHERE clause and without phpcs:ignore comment
- `SELECT * FROM table` followed by `array_filter()` in PHP
- Repository method loading all rows when only subset needed

## Why This Matters
- **Performance**: Full scans slow down as table grows (100 rows OK, 100k rows NOT OK)
- **Memory**: Loading entire table into PHP memory can cause OOM
- **Scalability**: Query that works in dev (100 rows) fails in prod (1M rows)

## When Full Scan Is OK
- Small lookup tables (< 1000 rows, rarely changes) — e.g., categories, settings
- Building hierarchical tree structure (categories for menu)
- Admin export/report features (with streaming, not loading all into memory)
