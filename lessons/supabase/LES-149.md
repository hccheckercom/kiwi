## Bad (actual state in webstore-vn)
```
supabase/migrations/
├── 003_align_schema.sql
├── 003_blog.sql              ← DUPLICATE prefix!
├── 004_atomic_coupon.sql
├── 004_blog_view_counter.sql ← DUPLICATE prefix!
├── 007_leads_table.sql
├── 007_revenue_aggregate.sql ← DUPLICATE prefix!
```

## Good
```
supabase/migrations/
├── 003_align_schema.sql
├── 004_blog.sql
├── 005_atomic_coupon.sql
├── 006_blog_view_counter.sql
├── 007_leads_table.sql
├── 008_revenue_aggregate.sql
```

## Impact
- Local: both run (alphabetical order within same prefix)
- Supabase CLI push: unpredictable — may skip one
- New dev setup: different schema than production
- Result: "column X does not exist" errors that only happen in some environments

## Why
Real issue in webstore-vn migrations/. Supabase migrations run in filename order. Duplicate prefixes cause unpredictable execution — local DB may have both tables but production only has one. This is the root cause of "works on my machine" schema bugs.
