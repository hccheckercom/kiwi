## Bad
```php
// AnalyticsCollector.php:31
$ua = sanitize_text_field( wp_unslash( substr( $context['user_agent'] ?? ( $_SERVER['HTTP_USER_AGENT'] ?? '' ), 0, 512 ) ) );

// CheckoutController.php:108 â€” GOOD example
$ip = explode( ',', sanitize_text_field( wp_unslash( $_SERVER[ $header ] ) ) );
```

## Good
```php
// ALWAYS wp_unslash() before sanitize on $_SERVER, $_GET, $_POST, $_COOKIE
$ua = isset( $_SERVER['HTTP_USER_AGENT'] ) 
    ? sanitize_text_field( wp_unslash( substr( $_SERVER['HTTP_USER_AGENT'], 0, 512 ) ) )
    : '';

// Or use null coalescing with wp_unslash
$ua = sanitize_text_field( wp_unslash( $_SERVER['HTTP_USER_AGENT'] ?? '' ) );
```

## Why
WordPress may add slashes to superglobals via `wp_magic_quotes()`. Without `wp_unslash()`:
- Backslashes leak into database (`O\'Reilly` instead of `O'Reilly`)
- Validation bypasses (e.g., `\'` passes email validation but breaks queries)
- Security issues when slashes affect SQL escaping

**Always** call `wp_unslash()` before sanitizing `$_SERVER`, `$_GET`, `$_POST`, `$_COOKIE`.

## Grep
`\$_(?:SERVER|GET|POST|COOKIE)\[.*?\](?!.*wp_unslash)` — in `**/*.php`
