## Pattern

Static utility methods that modify WordPress options, database state, or plugin settings are callable from anywhere without capability checks. Unlike REST endpoints (covered by LES-323), these methods bypass permission_callback and can be invoked by any code — including user-facing forms, AJAX handlers, or third-party plugins.

## Scan


## Bug

```php
// SAI — any code can call Commerce::disable('products')
class Commerce {
    public static function disable( string $module ): void {
        if ( in_array( $module, self::MODULES, true ) ) {
            update_option( "wezone_commerce_{$module}_enabled", false, false );
        }
    }
    
    public static function enable( string $module ): void {
        if ( in_array( $module, self::MODULES, true ) ) {
            update_option( "wezone_commerce_{$module}_enabled", true, false );
        }
    }
}

// Attacker code trong theme functions.php hoặc malicious plugin
add_action( 'init', function() {
    Commerce::disable( 'products' ); // Tắt toàn bộ products module!
});
```

**Hậu quả:**
- Malicious plugin/theme có thể disable critical modules
- User-facing AJAX handler gọi method này → privilege escalation
- No audit trail — không biết ai đã thay đổi settings

## Fix

### Option 1: Add capability check inside method (recommended)

```php
// ĐÚNG — check capability trước khi modify state
class Commerce {
    public static function disable( string $module ): void {
        if ( ! current_user_can( 'manage_options' ) ) {
            return; // hoặc throw exception
        }
        
        if ( in_array( $module, self::MODULES, true ) ) {
            update_option( "wezone_commerce_{$module}_enabled", false, false );
        }
    }
    
    public static function enable( string $module ): void {
        if ( ! current_user_can( 'manage_options' ) ) {
            return;
        }
        
        if ( in_array( $module, self::MODULES, true ) ) {
            update_option( "wezone_commerce_{$module}_enabled", true, false );
        }
    }
}
```

### Option 2: Make methods private, expose via controlled interface

```php
// ĐÚNG — private methods, public API có capability check
class Commerce {
    private static function disable( string $module ): void {
        if ( in_array( $module, self::MODULES, true ) ) {
            update_option( "wezone_commerce_{$module}_enabled", false, false );
        }
    }
    
    // Public API với capability check
    public static function admin_disable_module( string $module ): bool {
        if ( ! current_user_can( 'manage_options' ) ) {
            return false;
        }
        self::disable( $module );
        return true;
    }
}
```

### Option 3: Document as internal API, add nonce requirement

```php
// ĐÚNG — require nonce + capability
class Commerce {
    /**
     * INTERNAL API — only call from admin context with verified nonce.
     */
    public static function disable( string $module ): void {
        // Require both capability AND nonce
        if ( ! current_user_can( 'manage_options' ) ) {
            wp_die( 'Unauthorized', 403 );
        }
        
        if ( ! check_admin_referer( 'wezone_commerce_settings' ) ) {
            wp_die( 'Invalid nonce', 403 );
        }
        
        if ( in_array( $module, self::MODULES, true ) ) {
            update_option( "wezone_commerce_{$module}_enabled", false, false );
        }
    }
}
```

## Áp dụng

**Mọi public static method modify state (options, DB, files) PHẢI:**
1. Check `current_user_can()` với appropriate capability
2. Hoặc: make method private, expose via controlled public API
3. Hoặc: require nonce verification (nếu gọi từ admin forms)

**Exceptions:**
- Read-only methods (`isEnabled()`, `get*()`) — không cần capability check
- Methods chỉ gọi từ hooks với known context (document rõ ràng)
- Internal helper methods (mark `private` hoặc `@internal`)
