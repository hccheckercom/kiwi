## Bad
```ts
export async function handlePayment(order: Order) {
  await updateOrderStatus(order.id, 'paid')
  await sendTelegramMessage(buildNotify(order)) // Telegram down → payment handler fails!
  return Response.json({ success: true })
}
```

## Good (fire-and-forget pattern from webhook.handler.ts)
```ts
export async function handlePayment(order: Order) {
  await updateOrderStatus(order.id, 'paid')

  // Fire-and-forget — never block critical path
  try { fireWebhookTelegram(order, orderCode, newStatus, amount) }
  catch (e) { console.error('[Telegram] Build message failed:', e) }

  return Response.json({ success: true })
}
```

## Telegram function pattern (lib/telegram.ts)
```ts
export async function sendTelegramMessage(text: string): Promise<void> {
  const token = process.env.TELEGRAM_BOT_TOKEN
  const chatId = process.env.TELEGRAM_CHAT_ID
  if (!token || !chatId) return // Not configured — skip silently

  try {
    await fetch(`https://api.telegram.org/bot${token}/sendMessage`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ chat_id: chatId, text, parse_mode: 'HTML' }),
      signal: AbortSignal.timeout(5000), // 5s max — never block indefinitely
    })
  } catch (err) {
    console.error('[Telegram] Error:', err) // Log but don't throw
  }
}
```

## HTML escaping (prevent Telegram parse errors)
```ts
function escapeTgHtml(s: string): string {
  return s.replace(/&/g, '&amp;').replace(/</g, '&lt;').replace(/>/g, '&gt;')
}
```

## Why
Real pattern from webhook.handler.ts + lib/telegram.ts. External notifications (Telegram, email) must NEVER block the critical path (payment processing, order creation). If Telegram API is slow/down, the payment webhook times out → gateway retries → duplicate processing. Pattern: (1) guard env vars, (2) AbortSignal.timeout(5000), (3) catch errors internally, (4) escape HTML for parse_mode.
