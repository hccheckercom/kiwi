## Bad
```ts
// Public product listing — no Cache-Control → CDN doesn't cache → every request hits DB
return Response.json({ data: products })

// Private order data — no Cache-Control → CDN may cache → other users see it
return Response.json({ data: orders })
```

## Good (use centralized cache headers from lib/api-cache.ts)
```ts
import { CACHE_LONG, CACHE_MEDIUM, CACHE_NONE } from '@/lib/api-cache'

// Public, rarely changes (categories, settings, banners)
return Response.json({ data: categories }, { headers: { ...corsHeaders(req), ...CACHE_LONG } })
// s-maxage=300, stale-while-revalidate=60

// Public, changes moderately (products, reviews)
return Response.json({ data: products }, { headers: { ...corsHeaders(req), ...CACHE_MEDIUM } })
// s-maxage=30, stale-while-revalidate=30

// Private or frequently changing (orders, cart, auth)
return Response.json({ data: orders }, { headers: { ...corsHeaders(req), ...CACHE_NONE } })
// no-store, no-cache, must-revalidate
```

## Cache tier reference
| Tier | Header | Use for |
|------|--------|---------|
| CACHE_LONG | s-maxage=300, swr=60 | Categories, settings, banners |
| CACHE_MEDIUM | s-maxage=30, swr=30 | Products, reviews |
| CACHE_SHORT | s-maxage=10, swr=10 | Flash sales, stock |
| CACHE_NONE | no-store | Orders, cart, auth, mutations |
| CACHE_STREAM | no-cache, X-Accel-Buffering: no | SSE/streaming |

## Why
Real pattern from lib/api-cache.ts. Without explicit Cache-Control: (1) public data hits DB on every request (wasted resources), (2) private data may be cached by CDN/proxy and served to wrong users. Centralized constants ensure consistency and make it easy to audit cache policy across all routes.
